<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| daemon process → plugin process (Unix-socket IPC) | The daemon writes ChatDeliveryRequest envelope; the plugin parses it via Zod. Same-user, FS-permission gated socket (`~/.aof/data/daemon.sock` mode 0600). Trust is "local user processes." |
| plugin process → daemon process (Unix-socket IPC) | Plugin POSTs PluginCapabilityPost + ChatDeliveryResultPost. Same trust model as above. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-45-02-01 | T (Tampering) | ChatDeliveryRequest.systemEvent.sessionKey arrives over IPC; if tampered, could enqueue events into a victim session | mitigate | The daemon's notifier (Wave 3) sources `sessionKey` from `delivery.sessionKey` which Phase 44 captured from `OpenClawToolInvocationContextStore` at dispatch time (NOT from attacker-controlled task input). Wave 3 task action MUST trace `sessionKey` provenance back to invocation context, never to task frontmatter. Document this invariant in code comments. |
| T-45-02-02 | I (Information disclosure) | sessionKey appears in IPC envelope and structured logs | accept | sessionKey encodes `agent:<agentId>:<platform>:<chatId>:<topicId>` — chat IDs are not strictly secret in single-user dev; production deployments are single-user too (CLAUDE.md model). No additional redaction in Phase 45. |
| T-45-02-03 | S (Spoofing) | PluginCapabilityPost has `pluginId.default("openclaw")` — could a future second plugin spoof "openclaw" to override capability cache? | accept | Phase 45 has exactly one plugin. PluginCapabilityPost.pluginId is informational, not authentication. Document as a known future-extension point; revisit if multi-plugin lands. |
| T-45-02-04 | D (Denial of service) | Tampered request with very large `systemEvent.text` could fill OpenClaw's per-session queue (MAX_EVENTS=20 cap; 21st drops oldest) | accept | OpenClaw's bounded queue absorbs the abuse. Daemon-side rendering produces ~120-char text. Risk floor: a malicious daemon could DoS its own plugin — outside our trust model. |
</threat_model>
