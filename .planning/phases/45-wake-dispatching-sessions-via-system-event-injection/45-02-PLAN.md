<objective>
Wave 1 — extend the type surface and IPC schemas so Wave 2 (plugin-side) and Wave 3 (daemon-side) have a contract to compile against.

Three additive type/schema changes plus one load-bearing IPC-return-type change. The IPC-return-type change (Task 3) is the surgical fix for Checker Blocker #1: today the chat-delivery queue's awaiter resolves with `void`, so per-channel ACK data parsed from the plugin's POST body is dropped on the floor. Wave 3's notifier code (`deliverOne`) cannot emit per-channel telemetry against `undefined`. We widen the queue's `Waiter.resolve` typedef + `enqueueAndAwait` return type to carry `ChatDeliveryResultPost`, then propagate that through `MatrixMessageTool.send` and the daemon's `queueBackedMessageTool.send`. After this task, the round-trip wiring is real and the Wave 0 round-trip test (Plan 01 Task 1E new) flips RED → GREEN.

Per RESEARCH.md §"Architectural reality: daemon vs plugin": `runtime.system` only exists in the plugin process; the daemon-side notifier MUST forward call args to the plugin via IPC. This plan creates the wire contract for that forwarding AND the return-path wiring that brings the per-channel ACK back.

Per CLAUDE.md §Conventions: Zod is source of truth — `const Foo = z.object({...})` + `type Foo = z.infer<typeof Foo>`. Every new field carries `.describe()` (`npm run docs:generate` enforces). `.passthrough()` already covers forward-compat at the inner `delivery` block; outer envelope was permissive before — DO NOT add `.strict()` (would break older plugins).

Per Checker Blocker #2: the chat channel of `ChatDeliveryResultPost` uses `z.literal(true)` (no error field). Failure of the chat send is signalled to the daemon via a thrown exception (preserves Phase 44 agent-callback-fallback audit branch at `openclaw-chat-delivery.ts:184-227`), never via `result.chat.success: false`. Only systemEvent + heartbeat blocks have nullable `error` fields — those failures are explicitly non-fatal per D-45-CHANNEL-ORTHOGONALITY.
