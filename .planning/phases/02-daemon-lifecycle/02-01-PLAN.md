<objective>
Redesign the health endpoint to use a Unix socket with separate liveness (/healthz) and full status (/status) routes, fix the startup sequence so the PID file is only written after the health endpoint is confirmed working, and add crash recovery detection.

Purpose: The current health server binds to a TCP port (collision-prone, requires auth) and the PID file is written before the health server is ready. This plan fixes both issues, satisfying DAEM-03 (health binds before PID) and DAEM-04 (rich status endpoint), and lays the foundation for the service file generation in Plan 02.

Output: Unix socket health server, rich /status endpoint, PID-gated startup, crash recovery event emission.
</objective>

  <action>
**server.ts changes:**
Replace the TCP-based `createHealthServer(getState, store, port, bind)` with a Unix socket version: `createHealthServer(getState, store, socketPath)`. The server should:
- Listen on a Unix domain socket at `socketPath` (e.g. `~/.aof/daemon.sock`)
- Handle `GET /healthz` — call a new `getLivenessStatus()` that returns just `{ status: "ok" }` with HTTP 200 (or 503 if truly dead). This is the supervisor watchdog endpoint — minimal work, no store queries.
- Handle `GET /status` — call the existing `getHealthStatus()` which returns the full JSON with task counts etc.
- Delete the socket file if it already exists on startup (stale from crash). Use `unlinkSync` with try/catch.
- Export a new `selfCheck(socketPath)` function that makes a GET /healthz request to the socket and returns true/false. Use Node's `http.get` with `socketPath` option. This is used by daemon.ts to verify the server is working before writing the PID file.

**health.ts changes:**
- Add `version` field to `HealthStatus` — read from package.json via `createRequire(import.meta.url)('../../../package.json')` or a simpler approach: accept version as a parameter to `getHealthStatus()`.
- Add `components` field to `HealthStatus`: `{ scheduler: "running"|"stopped", store: "ok"|"error", eventLogger: "ok"|"error" }`.
- Add `config` field with summary: `{ dataDir: string, pollIntervalMs: number, providersConfigured: number }`. Accept these as additional parameters or via an expanded `DaemonState` interface.
- Add `getLivenessStatus()` export — returns `{ status: "ok" }` synchronously (no store queries). Only returns "error" if the process is in shutdown mode.
- Keep backward compat: `getHealthStatus()` signature can add optional params for the new fields.

**Test updates:**
- Update `server.test.ts` to use Unix socket instead of TCP port. Use `http.get({ socketPath, path: "/healthz" })` for requests in tests.
- Test that `/healthz` returns 200 with `{ status: "ok" }`.
- Test that `/status` returns full JSON with version, components, config, and task counts.
- Test that unknown routes return 404.
- Update `health.test.ts` to verify new fields (version, components, config) in the HealthStatus output.

Per user decision: Socket path naming is Claude's discretion. Use `join(dataDir, "daemon.sock")` as the default socket path.

<task type="auto">
  <name>Task 2: Fix startup sequence (PID gating + crash recovery) and add system.crash_recovery event</name>
  <files>src/daemon/daemon.ts, src/schemas/event.ts, src/daemon/__tests__/daemon.test.ts</files>
  <action>
**daemon.ts changes:**
1. Replace `healthPort` and `healthBind` options with `socketPath` in `AOFDaemonOptions`. Default: `join(opts.dataDir, "daemon.sock")`.

2. **Fix startup order** (DAEM-03): Currently PID file is written BEFORE `service.start()` and health server creation. Change to:
   a. Create AOFService (constructor only, no start)
   b. Start health server on Unix socket (binds immediately)
   c. Self-check: call `selfCheck(socketPath)` — if it fails, throw "Health server failed to start"
   d. Write PID file ONLY after self-check succeeds
   e. Start service (`service.start()` — begins polling)

3. **Crash recovery detection:** Before step (a), check for stale PID file. If PID file exists but process is NOT running:
   - Store `previousPid` for logging
   - Delete stale PID file
   - After service starts, log `system.crash_recovery` event with payload: `{ previousPid, recoveredAt: ISO timestamp }`
   - Console: `[AOF] Recovered from crash (previous PID: ${previousPid})`

4. **Event schema**: Add `"system.crash_recovery"` to the `EventType` enum in `src/schemas/event.ts`.

5. Update signal handlers to also clean up the socket file on exit (unlink socket path).

6. **Config forwarding fix** (noted in Phase 1 verification and Phase 2 CONTEXT.md specifics): Ensure `pollTimeoutMs` and `taskActionTimeoutMs` from `AOFDaemonOptions` are forwarded to the `AOFService` constructor config. Currently `startAofDaemon()` only passes `dataDir`, `dryRun`, `pollIntervalMs`, `defaultLeaseTtlMs` — add the timeout options.

**daemon.test.ts changes:**
- Update all tests to use `socketPath` instead of `healthPort`/`healthBind`.
- Add test: "writes PID file only after health server self-check succeeds" — mock selfCheck to resolve, verify PID file exists after startAofDaemon completes.
- Add test: "detects stale PID file and emits crash recovery event" — write a stale PID file (non-existent PID), call startAofDaemon, verify `system.crash_recovery` event was logged.
- Add test: "does not emit crash recovery on clean start" — no PID file exists, verify no crash_recovery event.
- Add test: "cleans up socket file on shutdown" — after drainAndExit, verify socket path is deleted.
