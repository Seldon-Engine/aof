**Rewrite the `daemonStatus()` function** in `src/cli/commands/daemon.ts`:

1. **Check daemon running:** Read PID file, check process exists. If not running:
   - Print: `Daemon is not running. Run \`aof daemon install\` to start.`
   - Exit code 2 (per user decision)

2. **Query health endpoint:** Make HTTP GET to `/status` on the Unix socket at `join(dataDir, "daemon.sock")`. Use Node's `http.get({ socketPath, path: "/status" })`. Parse JSON response.

3. **If `--json` flag:** Print the raw JSON response (pretty-printed with 2-space indent) and exit.

   Scheduler:  running

<task type="auto">
  <name>Task 2: Redesign stop command with drain progress and verify watchdog restart</name>
  <files>src/cli/commands/daemon.ts, src/daemon/__tests__/daemon-cli.test.ts</files>
  <action>
**Rewrite the `daemonStop()` function:**

1. Read PID file. If not running: print `Daemon is not running. Run \`aof daemon install\` to start.` with exit code 2.

2. Print: `Stopping daemon (PID {pid})...`

3. **Send SIGTERM** (triggers the drain protocol from Phase 1):
   ```
   Stopping daemon (PID 12345)...
     Sent SIGTERM, draining in-flight work...
   ```

4. **Poll for process exit** with drain countdown (per user decision):
   - Poll every 500ms, check if process still running
   - Every 2 seconds, print drain status: `  Draining... {N}s remaining`
   - Default timeout: 15 seconds (10s drain + 5s buffer)

5. **On clean exit:**
   ```
     Drain complete.
   Daemon stopped.
   ```
   Clean up stale PID file and socket file. Exit code 0.

6. **On timeout:** Send SIGKILL, clean up, print warning. Exit code 0 (stop succeeded even if forced).

7. **Update stop via OS supervisor:** When installed as a service, prefer using the OS supervisor to stop:
   - macOS: `launchctl unload {plistPath}` (temporary stop, not uninstall)
   - Linux: `systemctl --user stop ai.openclaw.aof`
   - Fall back to direct SIGTERM if supervisor command fails
   - Add `--force` flag to bypass supervisor and send SIGTERM directly

**Watchdog verification (DAEM-05):**
Add a dedicated test in `daemon-cli.test.ts` (or a comment/documentation block if the test requires a real launchd environment) that describes the watchdog verification procedure:

Since we cannot run launchd/systemctl in unit tests, add a **manual verification script** at `scripts/verify-watchdog.sh` that:
1. Runs `aof daemon install`
2. Verifies daemon is running via `aof daemon status`
3. Sends SIGKILL to the daemon PID
4. Waits up to 30 seconds
5. Checks if daemon is running again (OS supervisor should have restarted it)
6. Runs `aof daemon uninstall` to clean up
7. Exits 0 on success, 1 on failure

Also add a unit test that verifies the service file configuration enables watchdog behavior:
- Test that generated launchd plist has `KeepAlive: true` and `ThrottleInterval: 5`
- Test that generated systemd unit has `Restart=on-failure` and `RestartSec=5`
(These tests exist in Plan 02 but add assertions here that confirm DAEM-05 specifically: "the generated service files configure the OS to restart on failure")

**Exit code consistency:** Review all daemon commands and ensure:
- 0 = success
- 1 = general error
- 2 = daemon not running (for stop and status when daemon is down)
