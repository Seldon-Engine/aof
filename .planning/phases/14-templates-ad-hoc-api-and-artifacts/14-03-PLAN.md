  <files>src/store/task-store.ts, src/store/task-store.test.ts</files>
  <behavior>
    - Test: store.create() with workflow.definition auto-validates via validateDAG and auto-initializes state via initializeWorkflowState
    - Test: store.create() with invalid workflow.definition throws with validation errors
    - Test: store.create() without workflow field works as before (backward compat)
    - Test: store.create() with workflow.definition + workflow.templateName preserves templateName on resulting TaskWorkflow
    - Test: Resulting task frontmatter contains complete TaskWorkflow (definition + state + optional templateName)
  </behavior>
  <action>
    In src/store/task-store.ts:
    - Import `validateDAG`, `initializeWorkflowState` from `../schemas/workflow-dag.js`
    - In the `create()` method, add workflow handling BEFORE the TaskFrontmatter.parse() call:
      - If `opts.workflow?.definition` is present (ad-hoc or pre-resolved template):
        - Run `const dagErrors = validateDAG(opts.workflow.definition)`
        - If errors: `throw new Error(\`Workflow DAG invalid: \${dagErrors.join(', ')}\`)`
        - Compute `state = initializeWorkflowState(opts.workflow.definition)`
        - Set `opts.workflow = { definition: opts.workflow.definition, state, templateName: opts.workflow.templateName }` -- complete TaskWorkflow object
    - This handles both ad-hoc (agent writes definition directly) and template (CLI resolves and passes definition + templateName)

    Per research recommendation: Template resolution happens in the CLI command handler, NOT in store.create(). The store only receives a complete or partial workflow object and handles validation + init.

    Write tests that exercise both paths. Use minimal valid WorkflowDefinition fixtures (2-3 hops with simple dependency).
  </action>
  <verify>
    <automated>npx vitest run src/store/task-store.test.ts --reporter=verbose 2>&1 | tail -30</automated>
  </verify>
  <done>store.create() auto-validates and auto-initializes workflow DAGs. Invalid DAGs throw. Both ad-hoc and template-resolved workflows produce identical TaskWorkflow runtime objects.</done>
