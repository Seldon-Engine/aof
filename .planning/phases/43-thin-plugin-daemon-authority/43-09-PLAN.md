  <how-to-verify>
    Prerequisites:
    1. Pull latest main on a developer Mac (or Linux box).
    2. `npm run deploy` — installs dist to `~/.aof` and symlinks the plugin.
    3. `aof setup --auto --upgrade` — runs all migrations including 007. Confirm output includes:
       - `007-daemon-required installed aof-daemon service` (or "already installed" if pre-existing daemon).
    4. `launchctl list | grep ai.openclaw.aof` (macOS) or `systemctl --user list-units | grep aof` (Linux) — daemon IS present and loaded.
    5. `aof daemon status` — reports running, shows uptime, socket path.

    Test A — tool invoke round-trip via plugin:
    1. Open an OpenClaw session (e.g., in VS Code or `openclaw chat`).
    2. Invoke `aof_status_report` tool via the agent — e.g., ask "show me my AOF task status".
    3. Observe: response renders correctly; daemon log (`~/.aof/data/logs/daemon.log`) shows a `/v1/tool/invoke` request with `name=aof_status_report`.
    4. Plugin log shows `DaemonIpcClient singleton initialized` once (not per session).

    Test B — dispatch + spawn round-trip:
    1. From an OpenClaw session, `aof_dispatch` a simple test task (e.g., `{ agentId: "<self>", title: "test", body: "smoke" }`).
    2. Daemon receives the IPC call, enqueues, picks up on poll, posts to /v1/spawns/wait.
    3. Plugin's spawn-poller receives the SpawnRequest, invokes `runEmbeddedPiAgent`.
    4. Task transitions: ready → in-progress → done (or review, depending on agent completion).
    5. Daemon log shows `spawn enqueued for plugin` then `spawn result received`.

    Test C — OpenClaw session reload:
    1. Close the current OpenClaw session, open a new one (triggers plugin reload).
    2. Dispatch a second task.
    3. Observe: daemon log shows ONE continuous `plugin attached` / NO `plugin detached` + `plugin attached` churn (singleton survived).
    4. Task dispatched via the SAME daemon process (`daemon.pid` unchanged).

    Test D — daemon crash + restart (D-03):
    1. `kill -9 $(cat ~/.aof/data/daemon.pid)` (macOS)
    2. Wait ~5s; launchd respawns.
    3. `aof daemon status` — running again, new PID.
    4. Plugin's spawn-poller reconnects within ~30s (exponential backoff).
    5. Dispatch a third task. Flow completes.

    Test E — --force-daemon deprecation:
    1. From a tarball: `sh install.sh --force-daemon`.
    2. Output contains: `--force-daemon is deprecated — the daemon is always installed in plugin-mode as of v1.15. Flag will be removed in a future release.`
    3. Daemon still installs (no behavior difference).

    Report:
    - PASS: all 5 tests complete as described.
    - FAIL: describe which test failed, what log output showed, which PLAN to amend.
  </how-to-verify>
