<task type="auto" tdd="true">
  <name>Task 1: Trace schema and session parser</name>
  <files>src/schemas/trace.ts, src/trace/session-parser.ts, src/trace/__tests__/session-parser.test.ts, tests/fixtures/session-basic.jsonl, tests/fixtures/session-debug.jsonl</files>
  <behavior>
    - parseSession(filePath, { debug: false }) extracts tool calls with name + truncated input (200 chars), model, provider, thinkingLevel from a valid JSONL fixture
    - parseSession(filePath, { debug: true }) includes full tool input, full tool output (from toolResult messages), and reasoning text from thinking content
    - parseSession with malformed lines increments parseErrors counter, never throws
    - parseSession with unknown entry types increments unknownEntries counter
    - parseSession with missing file returns a result with empty toolCalls and zero counts (does not throw)
    - Summary mode: reasoning array is undefined/empty
    - Debug mode: reasoning array contains thinking text from assistant messages
    - model_change entries populate session.model and session.provider
    - thinking_level_change entries populate session.thinkingLevel
    - Tool calls handle both "toolCall" and "tool_use" content types
    - toolResult messages are matched to tool calls by toolCallId and populate the output field (debug only)
  </behavior>
  <action>
    **1. Create Zod schema (`src/schemas/trace.ts`):**

    Define the trace schema with these shapes (per CONTEXT.md: same schema shape for both modes, debug just has more data):

    ```
    ToolCallTrace: { name: string, input: string, output?: string, toolCallId?: string }
    TraceSchema: {
      version: 1 (literal),
      taskId: string,
      sessionId: string,
      attemptNumber: number (positive int),
      capturedAt: string (datetime),
      session: { sessionFilePath: string, durationMs: number, model?: string, provider?: string, thinkingLevel?: string },
      toolCalls: ToolCallTrace[],
      toolCallCount: number,
      reasoning?: string[],  // debug mode only
      noopDetected: boolean,
      meta: { mode: "summary" | "debug", unknownEntries: number (default 0), parseErrors: number (default 0), truncated: boolean (default false), totalEntriesParsed: number }
    }
    ```

    Export `TraceSchema`, `ToolCallTrace`, and their inferred TypeScript types.

    **2. Create test fixtures:**

    - `tests/fixtures/session-basic.jsonl`: A session with 1 session entry, 1 model_change, 1 thinking_level_change, 1 user message, 1 assistant message with 1 toolCall + 1 thinking block, 1 toolResult. Known values for deterministic assertions.
    - `tests/fixtures/session-debug.jsonl`: Same structure but with longer tool inputs (>200 chars) and substantial thinking text, to test truncation in summary mode vs full capture in debug mode.

    **3. Create session parser (`src/trace/session-parser.ts`):**

    Export `parseSession(filePath: string, opts: { debug: boolean }): Promise<ParsedSession>` where ParsedSession contains the extracted data (toolCalls, model, provider, thinkingLevel, reasoning, unknownEntries, parseErrors, totalEntriesParsed).

    Use `node:readline` with `createReadStream` for line-by-line streaming. Parse each line in try/catch independently. Switch on `entry.type`:
    - `"session"`: extract session metadata
    - `"model_change"`: store provider, modelId
    - `"thinking_level_change"`: store thinkingLevel
    - `"message"` with `role: "assistant"`: scan content array for `type: "toolCall"` or `type: "tool_use"`, extract name + input (truncate to 200 chars if not debug). Also scan for `type: "thinking"` and collect reasoning (debug only).
    - `"message"` with `role: "toolResult"`: match to tool call by toolCallId, store output (debug only).
    - `"custom"`: skip silently
    - default: increment unknownEntries

    For tool call input: if arguments/input is an object, JSON.stringify it. In summary mode, truncate to 200 chars. In debug mode, keep full.

    If file doesn't exist, return empty ParsedSession (toolCalls: [], parseErrors: 0, etc.) without throwing.

    **Important:** Handle both `type: "toolCall"` (with `arguments`) and `type: "tool_use"` (with `input`) content types per CONTEXT.md.
  </action>
  <verify>
    <automated>npx vitest run src/trace/__tests__/session-parser.test.ts --reporter=verbose</automated>
  </verify>
  <done>
    - TraceSchema validates correct trace objects and rejects malformed ones
    - parseSession extracts tool calls, model, provider, thinking level from fixture
    - Summary mode truncates inputs to 200 chars and excludes reasoning
    - Debug mode includes full inputs, outputs, and reasoning
    - Malformed lines and unknown types handled gracefully with counters
    - Missing file returns empty result without throwing
  </done>
</task>

  <behavior>
    - detectNoop with zero toolCalls returns { noopDetected: true }
    - detectNoop with one or more toolCalls returns { noopDetected: false }
    - detectNoop with sessionMissing: true returns { noopDetected: false, skipped: true } (per user decision: missing session = skip no-op detection)
  </behavior>
  <action>
    **1. Create test fixture:**

    - `tests/fixtures/session-noop.jsonl`: A session with session entry, model_change, user message, and assistant message with NO tool calls (just text content). This represents the original Phase 25 incident pattern.

    **2. Create no-op detector (`src/trace/noop-detector.ts`):**

    Export:
    ```typescript
    interface NoopResult {
      noopDetected: boolean;
      skipped?: boolean;  // true when session file was missing/unreadable
    }

    function detectNoop(opts: { toolCallCount: number; sessionMissing: boolean }): NoopResult
    ```

    Logic:
    - If `sessionMissing` is true: return `{ noopDetected: false, skipped: true }` (per user decision)
    - If `toolCallCount === 0`: return `{ noopDetected: true }`
    - Otherwise: return `{ noopDetected: false }`

    This is intentionally simple. The complexity is in the session parser, not the detector. "Meaningful tool call" = any tool call at all (per Phase 25 decision).
  </action>

  </verify>
  <done>
    - Zero tool calls detected as no-op
    - Non-zero tool calls not flagged
    - Missing session files skip detection entirely (not flagged as no-op)
  </done>
