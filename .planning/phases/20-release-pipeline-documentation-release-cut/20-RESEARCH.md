## Summary

Phase 20 is the final phase of the v1.3 milestone. It wires the existing `scripts/verify-tarball.mjs` (built in Phase 19) into the `release.yml` GitHub Actions workflow as a gate between tarball build and upload, writes an UPGRADING.md document for end users, and prepares the repository for the v1.3.0 release cut via the already-configured `release-it` toolchain.

All the building blocks exist: `verify-tarball.mjs` is a standalone script that exits non-zero on failure (tested in Phase 19), `release.yml` is a straightforward sequential workflow, `.release-it.json` is fully configured with conventional-changelog, and the three v1.3 migrations (001-default-workflow-template, 002-gate-to-dag-batch, 003-version-metadata) are implemented and tested. The work is purely additive: one new YAML step in a workflow, one new markdown file, and no code changes.

**Primary recommendation:** Add the verify-tarball step as a single `run:` line in `release.yml`, write UPGRADING.md covering all three upgrade paths (fresh, pre-v1.2, v1.2), and document the `release-it` invocation as a manual human step.

## Architecture Patterns

### Release Flow (End-to-End)

```
Developer runs:
  npm run release -- 1.3.0
    |
    v
release-it (local):
  1. before:init hooks: npm run typecheck && npm test
  2. Bump package.json version to 1.3.0
  3. Generate CHANGELOG.md via conventional-changelog
  4. git commit "chore: release v1.3.0"
  5. git tag v1.3.0
  6. git push origin main --tags
  7. Create GitHub Release (empty, no assets)
    |
    v
GitHub Actions release.yml (triggered by v* tag push):
  1. Checkout + setup Node 22
  2. npm ci
  3. Typecheck
  4. Build (tsc)
  5. Test (npm test)
  6. Extract version from GITHUB_REF_NAME (v1.3.0)
  7. Generate changelog body (from git log)
  8. Build release tarball: node scripts/build-tarball.mjs v1.3.0
  9. ** NEW: Verify tarball: node scripts/verify-tarball.mjs aof-v1.3.0.tar.gz **
  10. Upload tarball to GitHub Release (updates release-it's release)
```

### Tarball Naming Convention

The version variable in CI comes from `GITHUB_REF_NAME` which is `v1.3.0` (includes the `v` prefix). The `build-tarball.mjs` script creates `aof-${version}.tar.gz`, producing `aof-v1.3.0.tar.gz`. The installer downloads from `aof-v${VERSION}.tar.gz` where `VERSION` has the `v` stripped, producing `aof-v1.3.0.tar.gz`. These match correctly.

### Verify Step Integration

The verify step is a single `run:` line that receives the tarball path. If any of the 6 checks fail, the script calls `process.exit(1)`, which causes the GitHub Actions step to fail, which prevents subsequent steps (upload) from executing. This is standard GitHub Actions failure propagation -- no special configuration needed.

```yaml
- name: Verify release tarball
  run: node scripts/verify-tarball.mjs aof-${{ steps.version.outputs.version }}.tar.gz
```

### Anti-Patterns to Avoid
- **Manual CHANGELOG generation:** Let `release-it` handle CHANGELOG.md at tag time. Do not pre-generate or hand-write changelog entries.
- **Modifying install.sh:** The installer already handles upgrade paths and triggers migrations. No modifications needed.
- **Adding `aof smoke` to CI:** The smoke command is user-facing (post-install). The tarball verification script is the CI gate.

## Common Pitfalls

### Pitfall 1: release-it and softprops/action-gh-release Body Conflict
**What goes wrong:** Both `release-it` (via `"github.release": true`) and `softprops/action-gh-release` target the same GitHub Release for the same tag. The `softprops` action will overwrite the release body with the CI-generated changelog.
**Why it happens:** `release-it` creates the release first (when the developer runs `npm run release`), then CI triggers on the tag push and the `softprops` action updates the same release.
**How to avoid:** This is actually the intended behavior. The CI-generated changelog from `softprops/action-gh-release` using the body from the `changelog` step provides a consistent format. The `release-it` conventional-changelog writes to `CHANGELOG.md` file in the repo (which is the primary changelog artifact). The GitHub Release body is a secondary display.
**Warning signs:** If the GitHub Release body looks different from CHANGELOG.md, this is expected -- they use different formatters.

### Pitfall 2: Forgetting to Update package.json Version Before Tarball Build
**What goes wrong:** The tarball contains the old version in package.json, causing the verify-tarball version match check to fail.
**Why it happens:** The developer might think CI needs to bump the version, but `release-it` already bumped it locally and pushed the commit before tagging.
**How to avoid:** The flow is correct as-is: `release-it` bumps version -> commits -> tags -> pushes. CI checks out the tag which includes the bumped version. No issue.

### Pitfall 3: UPGRADING.md Referencing Non-Existent Commands
**What goes wrong:** Documentation references `aof rollback` or `aof migrations` which are deferred to v1.4+.
**Why it happens:** Confusion between current scope and future requirements.
**How to avoid:** Only reference `aof smoke`, `aof --version`, `aof setup --auto`, and manual snapshot restore. No CLI rollback command exists yet.

### Pitfall 4: Snapshot Path Confusion
**What goes wrong:** Documenting the wrong snapshot location for rollback.
**Why it happens:** Two backup mechanisms exist: (1) installer backups at `$INSTALL_DIR/.aof-backup/` (created by `install.sh`), and (2) migration snapshots at `$INSTALL_DIR/.aof/snapshots/` (created by the migration framework in `setup.ts`).
**How to avoid:** Document both mechanisms clearly. The `.aof/snapshots/` directory contains pre-migration snapshots created by the migration framework. The `.aof-backup/` directory contains broader data backups created by the installer before extraction.

### Migration Summary for UPGRADING.md Content

Three migrations run automatically during upgrade (for RELS-02 documentation):

1. **001-default-workflow-template** -- Adds `defaultWorkflow` field to each project's `project.yaml` (picks the first workflow template name). Projects without templates are unaffected.

2. **002-gate-to-dag-batch** -- Batch-converts all gate-based task frontmatter to DAG workflow format across all status directories. Tasks already in DAG format are skipped.

3. **003-version-metadata** -- Writes version tracking metadata to `.aof/channel.json` with `version`, `previousVersion`, `channel`, and timestamp.

### Rollback Information for UPGRADING.md Content

Two rollback mechanisms to document:

1. **Migration snapshots** (`.aof/snapshots/`): Created automatically before migrations run. Contains full copy of data directory. Last 2 snapshots retained. Manual restore requires copying files back.

2. **Installer backups** (`.aof-backup/`): Created by `install.sh` before tarball extraction during upgrades. Contains `tasks/`, `events/`, `memory/`, `state/`, `data/`, `logs/`, `Projects/`, `memory.db`, `memory-hnsw.dat`, `.version`. Automatic restore on extraction failure.
