<decisions>
## Implementation Decisions

### Pipeline smoke gate (RELS-01)
- CI-only gate: add a `verify-tarball.mjs` step in `release.yml` between "Build release tarball" and "Upload tarball to GitHub Release"
- If verify-tarball fails, the workflow fails and no release is published (standard GitHub Actions failure behavior)
- No additional `aof smoke` in CI — tarball verification is sufficient for release gating; `aof smoke` is a user-facing post-install tool
- No special failure notifications (Slack/Discord) — standard workflow failure with logs is sufficient
- Upgrade scenario tests already run as part of `npm test` — no separate CI step needed

### UPGRADING.md (RELS-02)
- Lives at repo root (`UPGRADING.md`), alongside README.md
- Audience: developers using AOF as a tool (not contributors/internals)
- Sections: What's New in v1.3, Prerequisites, Step-by-step upgrade, Rollback instructions
- Covers all upgrade paths: fresh install, upgrade from pre-v1.2, upgrade from v1.2
- Primary upgrade method: curl installer; secondary: manual tarball download
- Verification commands: `aof smoke`, `aof --version`
- Rollback: document snapshot restore from `.aof/snapshots/`

### Release cut process (RELS-03)
- Use `release-it` (already configured) to handle version bump, tagging, CHANGELOG.md generation
- Plan prepares everything (pipeline, docs) but stops at "ready to release" — actual `release-it` invocation is a human step
- CHANGELOG.md auto-generated by `@release-it/conventional-changelog` plugin at tag time
- No pre-generated changelog in the plan

### Installer
- No changes to `install.sh` needed — existing flow (download tarball, extract, npm ci, aof setup) already triggers migrations from Phase 17
- UPGRADING.md documents both installer and manual tarball paths

### Claude's Discretion
- Exact wording and structure of UPGRADING.md content
- release.yml step naming and ordering details
- Whether to add a "Verify release" checklist section to UPGRADING.md
- Any minor release.yml cleanup (e.g., consolidating steps)

</decisions>
