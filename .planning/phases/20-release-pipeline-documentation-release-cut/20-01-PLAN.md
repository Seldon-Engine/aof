Create `UPGRADING.md` at the repo root (alongside README.md). Audience: developers using AOF as a tool (not contributors/internals).

Per user decisions, the document MUST include these sections:

**What's New in v1.3:**
- DAG workflows replace gate-based workflows as the default task structure
- Automatic defaultWorkflow assignment for projects with workflow templates
- Migration framework with pre-migration snapshots for safe upgrades
- `aof smoke` post-install health check command
- Tarball verification in release pipeline
- Version tracking in `.aof/channel.json`

**Prerequisites:**
- Node.js >= 22 (LTS recommended)
- Existing AOF installation (if upgrading)

**Fresh Install:**
- Primary: `curl -fsSL https://raw.githubusercontent.com/d0labs/aof/main/scripts/install.sh | sh`
- Secondary: Manual tarball download from GitHub Releases, extract, `npm ci --omit=dev`, `aof setup --auto`

**Upgrading from v1.2:**
- Same curl installer command (detects existing installation, creates backup, extracts new version)
- Three migrations run automatically:
  1. 001-default-workflow-template: Adds `defaultWorkflow` field to project.yaml
  2. 002-gate-to-dag-batch: Converts all gate-based tasks to DAG workflow format
  3. 003-version-metadata: Writes version tracking to `.aof/channel.json`
- Verification steps after upgrade

**Upgrading from pre-v1.2:**
- Same process as v1.2 upgrade (installer handles both paths)
- Note: legacy data paths may require the installer to migrate more files
- All three migrations still apply
- Verification steps after upgrade

**Verification:**
- `aof --version` should show 1.3.0
- `aof smoke` runs 6 health checks: version, schema, task store, org chart, migration status, workflow templates

**Rollback:**
- Migration snapshots: `.aof/snapshots/` contains pre-migration copies of the data directory (last 2 retained). Restore by copying contents back to the data directory.
- Installer backups: `.aof-backup/` contains broader data backup created before tarball extraction. Restore by copying contents back.
- Document the manual restore process (no `aof rollback` CLI exists yet -- that is deferred to v1.4+). Do NOT reference `aof rollback` or `aof migrations` commands.

Keep the tone practical and direct. Use code blocks for commands. No promotional language.
