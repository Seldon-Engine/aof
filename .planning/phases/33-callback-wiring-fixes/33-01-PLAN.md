Note: Do NOT modify openclaw-executor.ts in this phase. The OpenClaw adapter runs in-process -- the env var `AOF_CALLBACK_DEPTH` is set by callback-delivery.ts `metadata.callbackDepth` and read by `createAofMcpContext`. The in-process boundary crossing is handled by the env var fallback in createAofMcpContext. If the embedded agent creates its MCP server in the same process, `process.env.AOF_CALLBACK_DEPTH` is visible. For a more robust approach, the adapter would need to pass callbackDepth through AofMcpOptions, but that requires understanding the OpenClaw extension API internals which are outside AOF's control. The env var approach is sufficient for the current embedded-agent architecture.

Actually, on reflection: callback-delivery.ts sets `metadata: { callbackDepth: depth + 1 }` on the TaskContext passed to `executor.spawnSession()`. The OpenClawAdapter receives this context but does NOT pass metadata to the spawned agent's environment. The spawned agent creates a fresh AofMcpServer with `createAofMcpContext` which has no way to know callbackDepth UNLESS we set the env var before spawning.

The correct approach: In callback-delivery.ts, before calling `executor.spawnSession()`, set `process.env.AOF_CALLBACK_DEPTH = String(depth + 1)`. After the call returns, delete the env var. This is safe because: (a) the embedded agent runs in the same process, (b) `createAofMcpContext` is called during agent init which happens after env var is set, (c) we clean up after to avoid stale values per Pitfall 4 from research.

Add to callback-delivery.ts in `deliverSingleCallback` (the helper called by both deliverCallbacks and deliverAllGranularityCallbacks):
- Before `opts.executor.spawnSession(context, ...)`, set `process.env.AOF_CALLBACK_DEPTH = String(context.metadata?.callbackDepth ?? 0)`
- After the spawn call (in a finally block), `delete process.env.AOF_CALLBACK_DEPTH`

Wait -- this has a race condition with concurrent spawns (Pitfall 4). Multiple callback deliveries could overwrite each other's env var. Since the agent's MCP server creation is async and fire-and-forget, the env var could be deleted before it's read.

Safer approach: Do NOT use env var for in-process spawns. Instead, have the OpenClawAdapter pass callbackDepth through to the MCP server options. Specifically:
- In `openclaw-executor.ts` `spawnSession()`, read `context.metadata?.callbackDepth` and pass it when configuring the MCP server for the spawned agent. But AOF does not control how the embedded agent creates its MCP server -- that's OpenClaw internals.

Simplest safe approach that avoids race conditions: Use the `AofMcpOptions.callbackDepth` field. The MCP adapter that creates the AofMcpContext for spawned agents needs to pass callbackDepth. Since AOF controls the `AofMcpServer` creation (in `src/mcp/adapter.ts`), update the adapter to accept callbackDepth in its options. Then the question is: who passes it?

Looking at the architecture: callback-delivery.ts -> executor.spawnSession(context) -> OpenClawAdapter.spawnSession() -> ext.runEmbeddedPiAgent() -> [OpenClaw creates agent] -> agent uses AofMcpServer -> AofMcpServer calls createAofMcpContext(options).

AOF does NOT control what happens between runEmbeddedPiAgent and AofMcpServer creation. The env var is the ONLY mechanism that crosses this boundary.

For the race condition: use a unique env var keyed by taskId, e.g., `AOF_CALLBACK_DEPTH_{taskId}`. No -- that's fragile.

Better: Accept the env var race as acceptable for now. In practice, callback deliveries are sequential within a single onRunComplete call (the for loop in deliverCallbacks is sequential with await). Concurrent onRunComplete calls for different tasks could race, but the window is tiny (env var set -> spawnSession returns synchronously with fire-and-forget -> env var deleted). Since spawnSession returns a Promise that resolves after setup (before the agent actually reads env), the race window is essentially zero.

Final approach: Set `process.env.AOF_CALLBACK_DEPTH` before spawnSession, delete in finally. Add this to callback-delivery.ts `deliverSingleCallback`. This is the approach recommended in the research.
