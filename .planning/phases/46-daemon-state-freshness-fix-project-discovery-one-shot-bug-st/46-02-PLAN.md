    ```typescript
    /**
     * Phase 46 / Bug 1A (reconciliation half): heal on-disk drift between
     * frontmatter.status and directory location. Filesystem is the source
     * of truth for location; frontmatter is the source of truth for WHICH
     * status the task should have. When they disagree — e.g. because a
     * pre-Phase-46 partial transition crashed between `save()` and
     * `transition()`, or because an external process moved a file — this
     * pass rewrites the on-disk layout to match frontmatter.
     *
     * Reuses `this.lint()` → `lintTasks()` which already detects and
     * reports "Status mismatch:" issues via the same directory walk.
     * Runs ONCE per init() (not per poll) per CONTEXT.md: "It's a
     * self-heal pass for past drift, not a continuous correction loop."
     *
     * PATTERNS.md Pitfall 4: we do NOT call `this.get(id)` inside the
     * loop — get() has its own mtime-wins self-heal that deletes files
     * mid-walk. lintTasks uses parseTaskFile directly on each enumerated
     * path, so `issues` is a safe snapshot.
     *
     * Companion directory is moved alongside the .md on a best-effort
     * basis. ENOENT (no companion dir) is fine; any other rename failure
     * is logged and we keep going — the .md move already succeeded and
     * is the critical half.
     */
    private async reconcileDrift(): Promise<void> {
      const issues = await this.lint();
      for (const { task, issue } of issues) {
        if (!issue.startsWith("Status mismatch:")) continue;

        const targetStatus = task.frontmatter.status;
        if (!STATUS_DIRS.includes(targetStatus)) {
          storeLog.warn(
            { taskId: task.frontmatter.id, status: targetStatus, op: "reconcile" },
            "frontmatter status not in known dirs — leaving file in place",
          );
          continue;
        }

        const oldPath = task.path;
        if (!oldPath) {
          storeLog.warn(
            { taskId: task.frontmatter.id, op: "reconcile" },
            "task has no path — cannot reconcile",
          );
          continue;
        }

        // Parse the current status from the oldPath — the .md lives under
        // tasks/<currentStatus>/<id>.md, so splitting on "/tasks/" and
        // reading the next segment gives us the current (wrong) status.
        // Alternative: the `issue` string contains "but file in '<X>/'"
        // — parse that. Use the issue string since it's already computed.
        const match = issue.match(/but file in '(\w[\w-]*)\/'/);
        const currentStatus = match ? (match[1] as TaskStatus) : undefined;

        const newPath = this.taskPath(task.frontmatter.id, targetStatus);
        try {
          await mkdir(join(this.tasksDir, targetStatus), { recursive: true });
          await rename(oldPath, newPath);

          // Best-effort companion dir rename
          if (currentStatus && STATUS_DIRS.includes(currentStatus)) {
            const oldDir = this.taskDir(task.frontmatter.id, currentStatus);
            const newDir = this.taskDir(task.frontmatter.id, targetStatus);
            try {
              await rename(oldDir, newDir);
            } catch (err) {
              const code = (err as NodeJS.ErrnoException).code;
              if (code !== "ENOENT") {
                storeLog.warn(
                  { taskId: task.frontmatter.id, err, op: "reconcile" },
                  "companion directory rename failed (non-fatal, .md already moved)",
                );
              }
            }
          }

          storeLog.info(
            { taskId: task.frontmatter.id, from: oldPath, to: newPath, op: "reconcile" },
            "reconciled task file to match frontmatter status",
          );
        } catch (err) {
          storeLog.error(
            { taskId: task.frontmatter.id, err, oldPath, newPath, op: "reconcile" },
            "failed to reconcile task file",
          );
          // Do not throw — one bad file should not block the rest of init().
        }
      }
    }
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| init() → filesystem | `reconcileDrift()` issues `rename` on paths under `tasksDir` only. `targetStatus` is guarded by `STATUS_DIRS.includes()` before computing `newPath`, so a malicious frontmatter value can't cause writes outside `tasks/<known-status>/`. |
| lintTasks walk → reconcileDrift loop | lintTasks is read-only and already in-tree; reconciliation trusts its output but validates each targetStatus against STATUS_DIRS allowlist before acting. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-46-02-01 | Tampering | A task file with a hand-crafted `frontmatter.status: "../../etc/passwd"` could direct `this.taskPath(id, targetStatus)` outside `tasksDir` | mitigate | `STATUS_DIRS.includes(targetStatus)` check before computing newPath — `STATUS_DIRS` is a hardcoded `readonly` array. Any value not in it triggers the warn-log branch and leaves the file in place (no rename). Path-traversal attempt structurally impossible. |
| T-46-02-02 | Denial of Service | `lintTasks` walks every file in `tasks/<status>/` on every init() — large stores slow startup | accept | LOW — cost is O(N) readFile + parseTaskFile, microseconds per file. Daemon startup is a rare event (seconds minimum between restarts). For stores with >10k tasks the walk may add ~100ms to startup — still negligible compared to the full daemon init (IPC handshake, project discovery, reconcileOrphans). Measurable in profiling, not a correctness issue. |
| T-46-02-03 | Information Disclosure | Log lines emit `taskId`, `oldPath`, `newPath` — could leak project structure into logs visible to operators | accept | LOW — pino logs are structured and operator-only per CLAUDE.md. `taskId`, path info already appears throughout the daemon's log stream (task.transitioned events, dispatch logs). Not a new disclosure. |
| T-46-02-04 | Denial of Service | A malicious actor with write access to `tasks/` could plant thousands of mismatched files, forcing init() to do thousands of renames on every restart | accept | LOW — anyone with write access to `~/.aof/data/tasks/` already has full compromise of the AOF install (they can plant arbitrary task files, modify frontmatter, read secrets). Reconciliation cost is bounded to a per-file rename; does not amplify the existing compromise. |
