<objective>
Bug 1A — Atomic save+transition. Today `transitionToDeadletter` does
`store.save(task)` + `store.transition(taskId, "deadletter")` as two separate
awaits. A crash, ENOSPC, or rename failure between them leaves the file in
`tasks/ready/` with `frontmatter.status: deadletter` — the exact split-state
that produced the 5 ghost tasks and 172 MB log spin-loop on 2026-04-24.

Fix: extend `TransitionOpts.metadataPatch` so `transitionTask` applies the
patch in the same critical section as the rename, then collapse the
failure-tracker call site to a single `store.transition({ metadataPatch })`
call. Reuses the existing per-task `TaskLocks` mutex (no new locking
primitive). Reuses the post-`6fbcb18` v1.16.3 write-new-then-delete-old
hardening (no rebuild of crash-recovery logic).

Purpose: eliminate the partial-state window forever; the BUG-005 stamp
behavior must be preserved.

Output: extended `ITaskStore.transition` contract, extended
`TransitionOpts`, collapsed `transitionToDeadletter`, and a regression test
proving (a) rollback on rename failure and (b) atomic stamp+rename on
success.
</objective>
