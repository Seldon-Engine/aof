<objective>
Bug 1C — Bounded log rotation. The 172 MB `daemon-stderr.log` incident on
2026-04-24 was caused by pino's unbounded write to `fd: 2`, captured by
launchd's `StandardErrorPath` redirect with no rotation. After 6 days of
churn the file was 970,735 lines and producing visible fsync pressure on
unrelated processes (Claude Code hooks).

Per CONTEXT.md addendum Q2, the fix MUST cover BOTH halves —
(a) wire `pino-roll` for pino's structured output, AND
(b) drop `fd: 2` from pino's destinations so launchd's stderr capture
becomes a rare-event channel (only Node-level uncaught crashes write to it).
Adding pino-roll alone without dropping fd:2 leaves the launchd-stderr.log
growth path intact.

Per CONTEXT.md addendum Q1, gzip-on-rotation is dropped. pino-roll@4 has no
native gzip support; 250 MB worst-case (5×50MB raw) is fine — bounded and
predictable. Gzip can be added later as an optimization without breaking
anything.

Mechanic: replace `pino.destination({ fd: 2, sync: false })` in
`src/logging/index.ts:21-34` with `pino.transport({ target: 'pino-roll',
options: { file: <dataDir>/logs/aof.log, size: '50m', limit: { count: 5 },
mkdir: true } })`. Update `resetLogger()` to call `.end()` on the worker
transport so the vitest worker thread is released (orphan-vitest-workers
hazard from CLAUDE.md).

Purpose: bounded log disk use (~250 MB worst-case) without operator
opt-in. The 172 MB incident becomes structurally impossible.

Output: pino-roll dependency installed, getRootLogger() wired to use it,
fd:2 dropped from the destination chain, regression test confirming the
config is correct.

    ```typescript
    /**
     * Pino-based structured logging factory.
     *
     * Creates child loggers with a bound `component` field.
     *
     * Phase 46 / Bug 1C: writes JSON to a rotated log file at
     * <dataDir>/logs/aof.log via pino-roll's worker-thread transport
     * (50 MB per file, 5 files retained, no gzip — 250 MB worst-case).
     *
     * Pre-Phase-46 the logger wrote to fd:2 (stderr), which launchd's
     * StandardErrorPath captured into daemon-stderr.log with no rotation.
     * After 6 days of churn that file was 172 MB / 970k lines and creating
     * visible fsync pressure. fd:2 is intentionally NOT a destination
     * here so launchd's stderr capture becomes a rare-event channel
     * (only Node-level uncaught crashes still write to it).
     *
     * Log level controlled by AOF_LOG_LEVEL via the config registry.
     *
     * @module logging
     */

    import pino, { type Logger, type DestinationStream } from "pino";
    import { join } from "node:path";
    import { getConfig } from "../config/registry.js";
    import { resolveDataDir } from "../config/paths.js";

    let root: Logger | null = null;
    let dest: DestinationStream | null = null;

    function getRootLogger(): Logger {
      if (root) return root;

      const { core } = getConfig();
      const logsDir = join(resolveDataDir(), "logs");

      // Phase 46 / Bug 1C: pino-roll worker-thread transport for bounded
      // log disk use. mkdir: true is mandatory — without it pino-roll
      // throws on first write inside the worker thread, silently dropping
      // every log line (RESEARCH.md Pitfall 1). size: 50m × count: 5
      // caps disk use at ~250 MB worst-case; gzip is intentionally not
      // enabled (pino-roll lacks native support; 250 MB raw is within
      // user-approved budget per CONTEXT.md addendum Q1).
      //
      // fd:2 is NOT a destination here. CONTEXT.md addendum Q2: leaving
      // pino on fd:2 would keep the launchd-captured daemon-stderr.log
      // growth path open. Drop fd:2 entirely; launchd's stderr capture
      // becomes a rare-event channel (Node-level uncaught crashes still
      // write to it via Node's default handler).
      dest = pino.transport({
        target: "pino-roll",
        options: {
          file: join(logsDir, "aof.log"),
          size: "50m",
          limit: { count: 5 },
          mkdir: true,
        },
      });

      root = pino(
        {
          level: core.logLevel,
          timestamp: pino.stdTimeFunctions.isoTime,
        },
        dest,
      );
      return root;
    }

    /**
     * Create a child logger with the given component name bound.
     *
     * The root logger is lazily initialized on first call, reading
     * the log level from `getConfig().core.logLevel`.
     */
    export function createLogger(component: string): Logger {
      return getRootLogger().child({ component });
    }

    /**
     * Reset logger singleton — for test isolation (mirrors resetConfig()).
     *
     * Flushes any buffered logs, then calls .end() on the worker-thread
     * transport so the worker is released. Without .end() the vitest
     * worker pool leaks threads (CLAUDE.md "Orphan vitest workers" hazard)
     * and `kill -9` becomes mandatory after every aborted run.
     */
    export function resetLogger(): void {
      if (dest) {
        if ("flushSync" in dest && typeof dest.flushSync === "function") {
          (dest as { flushSync: () => void }).flushSync();
        }
        // Phase 46: pino-roll transport is worker-backed; .end() releases
        // the worker so the test suite doesn't leak threads.
        if ("end" in dest && typeof (dest as { end: () => void }).end === "function") {
          (dest as { end: () => void }).end();
        }
      }
      root = null;
      dest = null;
    }

    export type { Logger } from "pino";
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| daemon process → pino-roll worker thread | pino-roll spawns a Node worker thread; logger output crosses the thread boundary. Worker is in-process, no network or IPC. |
| pino-roll worker → filesystem | Writes to `<dataDir>/logs/aof.log` (and rotated siblings). Path is constructed from `resolveDataDir()` — operator-controlled but trusted in this threat model. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-46-04-01 | Information Disclosure | Log files contain full task content (titles, briefs, metadata, error messages). 250 MB of operator-readable data on disk. | accept | Logs are operator-only (`~/.aof/data/logs/` permissions inherit from data dir, typically 0755 owner-readable). No new disclosure surface vs the pre-Phase-46 launchd-stderr.log behavior. Same data, just bounded. |
| T-46-04-02 | Denial of Service | A log-flood attacker could fill the rotation cycle (50 MB × 5) and force older logs to roll off, hiding evidence | accept | LOW — anyone able to flood AOF logs already has tool-invocation access (which is what generates log volume). Compromise is upstream of the log layer. Bounded rotation is MITIGATION not regression vs pre-Phase-46 unbounded. |
| T-46-04-03 | Tampering | An attacker with filesystem write access to `<dataDir>/logs/` could pre-create or symlink the rotated files to elsewhere | accept | LOW — filesystem write access to `<dataDir>` is already full compromise. Standard Unix file permissions on the data dir are operator's responsibility. |
| T-46-04-04 | Denial of Service | Worker-thread transport leaks under SIGTERM during rotation handoff could lose the last few log lines on shutdown | accept | LOW — pino documents `transport.end()` as graceful; at-most-a-few-lines loss on shutdown is acceptable per RESEARCH.md Assumption A1. The orphan-vitest-workers fix in resetLogger() addresses the test-suite version of this concern. |
| T-46-04-05 | Repudiation | Dropping fd:2 means Node-level uncaught crashes during pino-roll worker init may not be persisted anywhere | mitigate | LOW — Node's default uncaughtException handler still writes to stderr (default Node behavior, not pino's), and launchd's StandardErrorPath capture remains in place for that channel. The risk is bounded to the brief window during pino-roll worker init (milliseconds). Future: if observed in practice, add `process.on('uncaughtException', e => fs.appendFileSync(...))` fallback. Not blocking. |
</threat_model>
