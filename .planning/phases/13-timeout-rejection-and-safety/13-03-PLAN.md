  <name>Rejection Cascade and Circuit Breaker</name>
  <files>src/dispatch/dag-evaluator.ts, src/dispatch/__tests__/dag-rejection.test.ts, src/dispatch/dag-transition-handler.ts, src/dispatch/__tests__/dag-transition-handler.test.ts, src/protocol/router.ts</files>
  <behavior>
    **mapRunResultToHopEvent — rejection outcome mapping:**
    - Given runResult.outcome = "needs_review" AND the hop has canReject = true → outcome = "rejected"
    - Given runResult.outcome = "needs_review" AND the hop has canReject = false → outcome = "complete" (reviewer cannot reject, treat as done — existing behavior)
    - Given runResult.outcome = "done" → outcome = "complete" (unchanged)
    - Given runResult.outcome = anything else → outcome = "failed" (unchanged)
    - NOTE: mapRunResultToHopEvent needs access to the hop definition to check canReject. Update its signature to accept the Hop definition as a parameter.

    **evaluateDAG — rejection cascade:**
    - Given event.outcome = "rejected" AND rejectionStrategy = "origin":
      - Increment rejectionCount on the rejected hop
      - If rejectionCount >= maxRejections (default 3): fail the hop, cascade skips downstream (circuit-breaker)
      - Otherwise: reset ALL hops to pending (root hops with no dependsOn → "ready"), clear result/startedAt/completedAt/agent/correlationId on each, preserve rejectionCount on the rejected hop ONLY
      - Return readyHops = all root hops (ready for re-dispatch)
      - DAG status stays "in-progress" (not complete)

    - Given event.outcome = "rejected" AND rejectionStrategy = "predecessors":
      - Increment rejectionCount on the rejected hop
      - If rejectionCount >= maxRejections: fail the hop, cascade skips downstream (circuit-breaker)
      - Otherwise: reset ONLY the rejected hop + its immediate dependsOn predecessors
      - Hops being reset: clear result/startedAt/completedAt/agent/correlationId, set to "pending" or "ready" based on whether their own dependencies are satisfied
      - Completed parallel branches that are NOT in the reset set → stay done
      - Preserve rejectionCount on the rejected hop through reset
      - Return readyHops = reset hops that are now "ready" (their dependencies satisfied)

    - Given event.outcome = "rejected" AND rejectionStrategy is undefined/missing → default to "origin"

    **Circuit-breaker:**
    - Default maxRejections = 3 (hop.maxRejections is not on schema — use a constant DEFAULT_MAX_REJECTIONS = 3)
    - When rejectionCount >= DEFAULT_MAX_REJECTIONS:
      - Set rejected hop to "failed" with completedAt
      - Cascade skips to downstream hops (reuse existing cascadeSkips)
      - Log "dag.hop_rejection_cascade" with reason "circuit_breaker"
      - Check DAG completion (may now be failed)

    **handleDAGHopCompletion — rejection integration:**
    - After evaluateDAG returns with rejection changes, log "dag.hop_rejected" event with rejection notes (from runResult.notes)
    - When rejection cascade resets hops, the readyHops from evaluateDAG are returned for dispatch in the same poll cycle
    - reviewRequired = false on rejection (rejection IS the review result)

    **handleSessionEnd (router.ts) — rejection flow:**
    - The existing handleSessionEnd already calls handleDAGHopCompletion which returns readyHops
    - The poll cycle dispatch section already picks up ready hops for dispatch
    - No changes needed in router.ts for the basic flow — rejection produces ready hops that get dispatched in the next poll cycle
    - BUT: check if handleSessionEnd needs to handle the rejection event type differently for task status. Currently, when dagComplete=true, it sets task status. For rejection, dagComplete=false, so task stays in-progress. This should work as-is.

    **Edge cases:**
    - Rejection on a root hop (no predecessors) with predecessors strategy: only the rejected hop itself resets
    - Rejection on a hop with all predecessors already complete: predecessors reset to ready (their own deps are satisfied from other completed paths — wait, they get reset so their results are cleared. Re-check: in predecessors strategy, the predecessors of the rejected hop get reset. Those predecessors may have their own dependencies. For the reset predecessor, check if its dependsOn are all outside the reset set and complete → "ready". Otherwise → "pending".)
    - rejectionCount = 0 initially (not set on HopState) → treat as 0
    - Rejection on a hop where canReject = false should never reach evaluateDAG as "rejected" (caught in mapRunResultToHopEvent)
  </behavior>

   ```
   if (event.outcome === "rejected") {
     const hopDef = definition.hops.find(h => h.id === event.hopId)!;
     const currentCount = (newState.hops[event.hopId]?.rejectionCount ?? 0) + 1;
     const maxRejections = DEFAULT_MAX_REJECTIONS; // 3

     if (currentCount >= maxRejections) {
       // Circuit-breaker: fail the hop
       newState.hops[event.hopId] = {
         ...newState.hops[event.hopId]!,
         status: "failed",
         completedAt: timestamp,
         rejectionCount: currentCount,
       };
       changes.push({ hopId: event.hopId, from: "dispatched", to: "failed", reason: "circuit_breaker" });
       // Cascade skips downstream
       cascadeSkips(definition, newState, changes, downstreamIndex, event.hopId, timestamp);
     } else {
       // Apply rejection strategy
       const strategy = hopDef.rejectionStrategy ?? "origin";
       if (strategy === "origin") {
         resetAllHopsForOrigin(definition, newState, changes, event.hopId, currentCount, timestamp);
       } else {
         resetPredecessorHops(definition, newState, changes, event.hopId, currentCount, timestamp);
       }
     }

     // Determine ready hops after reset
     const readyHops = determineReadyHops(definition, newState);
     for (const hopId of readyHops) { ... }

     // Check DAG completion
     const dagStatus = checkDAGCompletion(newState);
     // ... return result
   }
   ```
