## Gray Area 2 — Envelope shape

## Gray Area 3 — Mismatch observability

| Option | Description | Selected |
|--------|-------------|----------|
| log.info only | Skip with structured logger only. Cheapest. Loses telemetry / dashboard affordance. | |
| EventLogger event only | New `scheduler_action_skipped` event in events.jsonl. Telemetry-friendly, harder for human ops to grep. | |
| Both | log.info AND `scheduler_action_skipped` event. Mirrors existing `scheduler_action_failed` precedent. | ✓ |

**Resolution:** Both. Cost is negligible (one extra `logger.log` call per skip, wrapped in best-effort try/catch like every other event-logger call in the codebase). The 2026-04-15 incident was diagnosed retroactively from logs — having a structured event makes future incidents easier to correlate. Matches Phase 46's telemetry-heavy rollout posture (we just added `pino-roll` rotation precisely so we can be louder during rollouts).
**Event payload locked:** `{ handler, expected, actual, reason }` where `reason ∈ { "status_mismatch", "leaseAgent_mismatch", "lastTransitionAt_advanced", "task_gone" }`.

## Gray Area 4 — Lock interaction (TOCTOU closure)

| Option | Description | Selected |
|--------|-------------|----------|
| Outside the lock | Precondition re-read happens before `lockManager.withLock`; only the mutation is locked. TOCTOU window remains. | |
| Inside the lock (handler-level wrap) | Wrap `(read → check → mutate)` in `lockManager.withLock(taskId, body)`. Standardizes the pattern handleExpireLease already uses. | ✓ |
| Push precondition into store.transition | Move the `expected` opt into `store.transition` itself. Cleanest architecturally — mutex already there. | |

**Resolution:** Handler-level wrap. The "push into store.transition" option is architecturally cleanest but (a) changes the public store interface, (b) couples store layer to scheduler-action semantics, (c) forces migration of all `store.transition` callers. Handler-level wrap matches the existing `handleExpireLease` pattern exactly and keeps the store layer's contract pristine.
**Open verification (deferred to research/plan):** confirm `InMemoryTaskLockManager.withLock` is re-entrant for the same taskId from within the same async chain. If non-re-entrant, the outer handler `withLock` + inner `store.transition` `withLock` (Fix A) would deadlock. Plan must include a lock-interaction test.
**Graceful degradation:** if `config.lockManager` is undefined (test opt-out), handlers run their body without the wrap. Behavior: precondition check still runs, just with the today-existing race window.
