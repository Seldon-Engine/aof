<objective>
Add two precondition guards to `handleStaleHeartbeat` (`src/dispatch/recovery-handlers.ts`) so it no-ops when its dispatch-time premise no longer holds. Specifically: short-circuit when the task's status is no longer `"in-progress"`, and short-circuit when `action.agent` is set but no longer matches the task's current `lease.agent`. Both guards return `{ executed: false, failed: false }` and emit a structured `log.info` via the existing `recovery-handlers` logger.

This is the design doc's "Phase 1 narrow patch" — the smallest change that makes the 2026-04-15 incident (TASK-2026-04-15-010) reproducible-as-test and fixed. It deliberately does NOT introduce: a `SchedulerAction.expected` envelope, a `precondition.ts` helper module, a `scheduler_action_skipped` Zod event type, scheduler-side population sites, or precondition guards on the four other state-mutating handlers. Those are deferred until/unless production evidence justifies the broader generalization.

Why narrow: production has shown the race ONCE, in ONE handler. The minimum fix is a status pre-check at the top of that one handler — ~10 LOC of source plus the regression test that reproduces the incident shape. Anything beyond that is preventative engineering for races we have not seen.

Output: handleStaleHeartbeat patched in place; one new bug-NNN regression test that goes RED→GREEN; two added unit tests in the existing recovery-handlers.test.ts. No schema changes, no new files, no new events.
</objective>

export async function handleStaleHeartbeat(

No structured EventLogger event is emitted — by design (see D-999.3-OBSERVABILITY-VIA-LOG). | T-999.3-01-02 | Denial of Service | Status check on every stale_heartbeat dispatch | accept | Two synchronous comparisons against already-loaded fields. O(1). No I/O. The pre-existing `store.get(action.taskId)` already happens; the new code reads `staleTask.frontmatter.status` and `staleTask.frontmatter.lease?.agent` — both already in memory. No amplification. |
| T-999.3-01-03 | Repudiation | Race window between precondition check and store.transition | accept | A check-then-act window remains in the handler body — assign-helpers could in principle race in between our check at line ~40 and the eventual `store.transition` further down. In practice this window is microseconds (synchronous run-loop ticks within one handler invocation). Fix A's per-task transition mutex inside `store.transition` (v1.14.8) prevents the duplicate-rename outcome even if the race fires. The narrow precondition guard closes 99% of the original 9ms queue→execute window; the residual sub-millisecond window is bounded by Fix A. Closing it fully would require either wrapping the body in `config.lockManager.withLock` (which doesn't help here — assign-helpers doesn't go through that lock) or pushing the precondition into store.transition (broader contract change). Out of scope; documented for the record. |
