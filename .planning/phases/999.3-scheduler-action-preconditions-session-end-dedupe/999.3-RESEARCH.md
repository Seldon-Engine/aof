# Phase 999.3: Scheduler action pre-condition envelope — session-end handler dedupe — Research

**Researched:** 2026-04-25
**Domain:** AOF dispatch chain (TypeScript, intra-process, ESM, vitest)
**Confidence:** HIGH (all open verifications resolved against current code)

## Summary

The locked CONTEXT.md decisions are technically achievable. The single highest-risk
question — re-entrancy of `withLock` when an outer wrap calls `store.transition`
internally — turns out to be a non-issue because there are **two separate lock
implementations**:

1. `InMemoryTaskLockManager` (`src/protocol/task-lock.ts`) — used by handlers via
   `config.lockManager`. **Strict promise-chain serialization, NOT re-entrant.**
2. `TaskLocks` (`src/store/task-lock.ts`) — instantiated *per FilesystemTaskStore*
   and used internally by `transition`/`cancel`/`update`/`addDep`/`removeDep`/
   `updateBody`. Independent of `InMemoryTaskLockManager`.

These two locks are separate `Map<string, Promise>` instances on different objects
and **do not share state**. An outer `config.lockManager.withLock(taskId, body)` that
calls `store.transition(taskId, …)` inside the body cannot deadlock against itself,
because the inner `store.transition` acquires the `FilesystemTaskStore.locks`
instance, not the outer `config.lockManager`. This is exactly what `handleExpireLease`
already does today (`lifecycle-handlers.ts:113-117` wraps the body with
`config.lockManager.withLock`, then calls `store.transition` inside — passing tests).

The only real re-entrancy hazard is **double-wrapping the same `lockManager`**. The
existing `executeAssignAction` already wraps its entire body with
`config.lockManager.withLock` at `assign-executor.ts:362-365`. Adding another
`config.lockManager.withLock` inside `handleAssign` would deadlock. The fix is to do
the precondition check *inside* `executeAssignAction`'s already-wrapped body,
not at the `handleAssign` boundary. This is a minor restructure, not a blocker.

The other big surprise: **hop-timeout actions (`scheduler.ts:202` →
`checkHopTimeouts`) emit only `type: "alert"` actions, not a `hop_timeout` type.**
They route to `handleAlert`, which is not in the in-scope handler list and does no
state mutation — `escalateHopTimeout` itself does the state mutation directly via
`store.save` *before* the action is queued. Hop-timeout is naturally out of scope
for this phase; no new handler needed.

**Primary recommendation:** Implement per CONTEXT.md decisions, with one structural
refinement — for `handleAssign`, perform the precondition check *inside*
`executeAssignAction` after the existing `withLock` wrap (around the `store.get`
that already happens at `assign-executor.ts:62`), rather than wrapping at the
`handleAssign` boundary. All other four handlers (`handleStaleHeartbeat`,
`handleExpireLease`, `handleSlaViolation`, `handlePromote`) get the wrap at the
handler boundary as the design doc describes.

## Open Verifications Confirmed

### V1. `withLock` re-entrancy (CRITICAL — answered)

**Finding:** `InMemoryTaskLockManager.withLock` is **NOT re-entrant**. The
implementation at `src/protocol/task-lock.ts:31-66` chains promises serially:
each call appends a new promise to `this.locks.get(taskId)` that only runs
*after* the previous one settles. A nested call from within the outer body would
queue waiting for the outer's promise to settle — which can only happen after
the inner returns. Classic deadlock by self-await.

**HOWEVER, this is not a problem for the proposed design:**

- `config.lockManager` (`InMemoryTaskLockManager`) is constructed in
  `src/service/aof-service.ts:114` and shared between `ProtocolRouter` and the
  scheduler config.
- `FilesystemTaskStore.transition` does NOT call `config.lockManager`. It uses
  its own `private readonly locks = new TaskLocks()` field at
  `task-store.ts:103`, instantiated independently per store.
- `TaskLocks` (`src/store/task-lock.ts:33`) is a different class with a
  different `inflight: Map<string, PromiseLike<unknown>>` storage.
- Therefore `config.lockManager.withLock(taskId, async () => store.transition(...))`
  never re-enters its own lock. The outer holds `lockManager.locks[taskId]`; the
  inner acquires `store.locks.inflight[taskId]`. No deadlock.

**Evidence:** `handleExpireLease` already does exactly this pattern today
(`lifecycle-handlers.ts:113-117`):
```ts
if (config.lockManager) {
  await config.lockManager.withLock(action.taskId, expireBody);
} else {
  await expireBody();
}
```
where `expireBody` calls `store.transition(action.taskId, "ready", …)`. The test
at `lifecycle-handlers.test.ts:142-154` ("wraps in lockManager when present")
confirms this works with a mock. The test at
`task-store-concurrent-transition.test.ts` confirms the inner `TaskLocks`
serialization works in isolation.

**The one real hazard — DOUBLE-WRAP:** `executeAssignAction` already wraps with
`config.lockManager.withLock` at `assign-executor.ts:362-365`. If we add another
`config.lockManager.withLock(action.taskId, …)` at the `handleAssign` boundary
(`lifecycle-handlers.ts:187`), the outer wrap will hold the lock and the inner
wrap will queue forever — **deadlock**.

**Mitigation (recommended):** Move `handleAssign`'s precondition check *inside*
`executeAssignAction`, after the existing `const latest = await store.get(action.taskId)`
at line 62. The function already re-reads the task there. The precondition check
becomes a small extension. This avoids double-wrapping and reuses the existing
read.

**Compatibility with CONTEXT.md D-999.3-LOCK:** D-999.3-LOCK says "All five
state-mutating handlers wrap their (read → check → mutate) body in
`config.lockManager.withLock(action.taskId, body)`." For four of the five
handlers this is straightforward — they don't currently wrap. For `handleAssign`,
the wrap is already in place at the executor level, one frame deeper. Result is
functionally identical (precondition check inside the lock) without the deadlock.
Document this in the plan: "handleAssign's precondition check happens inside
executeAssignAction's existing lockManager wrap, not at the handler boundary."
Compatible.

**Confidence: HIGH** — verified by reading both lock implementations, the
existing call sites, and the existing tests. (`src/store/task-lock.ts`, `src/protocol/task-lock.ts`,
`src/store/task-store.ts:103,587-597`, `src/dispatch/assign-executor.ts:362-365`,
`src/dispatch/lifecycle-handlers.ts:113-117`, `src/protocol/__tests__/task-lock.test.ts:11-37`).

### V2. Hop-timeout action handler (answered)

**Finding:** Hop-timeout actions are NOT in scope and need no new precondition
work, because `checkHopTimeouts` does NOT produce a `hop_timeout`-typed action.
It produces only `type: "alert"` actions which are routed to `handleAlert`,
which performs no state mutation.

**Evidence:** `escalation.ts:230-292` (`checkHopTimeouts`) iterates DAG hops and
calls `escalateHopTimeout`. `escalateHopTimeout` returns one of three shapes
(all with `type: "alert"`):
- `escalation.ts:71-77` (already-escalated path) → `{ type: "alert", … }`
- `escalation.ts:95-101` (no escalateTo configured) → `{ type: "alert", … }`
- `escalation.ts:206-213` (active escalation success or fallback) → `{ type: "alert", … }`

The actual state mutation for the active-escalation path happens inside
`escalateHopTimeout` itself at `escalation.ts:188` (`await store.save(task)`),
*before* the action is returned. The returned `alert` is just observability;
when it later reaches `handleAlert` (`alert-handlers.ts:18-35`) it's logged,
nothing else.

**Implication:** D-999.3-SCHEDULER-POPULATION's bullet for "Hop-timeout site
(`src/dispatch/scheduler.ts:202` → `checkHopTimeouts`) — populate matching the
hop's evaluated status" is moot. The hop-timeout site does not produce any of
the in-scope action types. No `expected` field needs to be populated by
`checkHopTimeouts`. Document this in the plan as "no-op for hop-timeout site —
hop-timeout produces only alert actions which don't carry preconditions."

The `escalateHopTimeout` direct `store.save` is itself a separate race-shape
concern (it reads task state at scan time and saves later), but it's out of
scope for this phase per CONTEXT.md (which doesn't mention DAG hops).

**Confidence: HIGH** — verified by reading every return statement in
`escalation.ts:40-213`. (`src/dispatch/escalation.ts`)

### V3. `handleAssign`'s lockManager interaction (answered)

**Finding:** Confirmed — `executeAssignAction` already wraps its body in
`config.lockManager.withLock` at `assign-executor.ts:362-365`. Inside the
wrapped body it calls `store.transition` (indirectly via `acquireLease` and
directly at line 292 `await store.transition(action.taskId, "blocked", …)`).
Since `store.transition` uses the separate `TaskLocks` (V1), the inner
transitions don't deadlock against the outer `lockManager` wrap. The setup
already works in production.

**Adding precondition at the handler level (not recommended):**
`handleAssign(...)` at `lifecycle-handlers.ts:187` is a thin delegation. Adding
`config.lockManager.withLock` here would double-wrap and deadlock as documented
in V1.

**Adding precondition inside `executeAssignAction` (recommended):**
At `assign-executor.ts:54-71` the function already does `const latest = await store.get(action.taskId)` and then `if (latest.frontmatter.status !== "ready") { … return }`. The precondition check from this phase is essentially a generalization of that existing check. Slot the new
`matchesExpected` call right there, returning the same `{ executed: false, failed: false }`
shape on mismatch but emitting the new structured skip log + event.

The existing dedup check at lines 68-71 already covers the `expected.status = "ready"`
case in spirit, but doesn't emit the new `scheduler_action_skipped` event. The new
implementation should:
1. Replace the existing `if (latest.frontmatter.status !== "ready")` and `if (isLeaseActive(latest.frontmatter.lease))` checks with the unified `matchesExpected(latest, action.expected)` check (or compose them).
2. On mismatch, emit `log.info` + `logger.log("scheduler_action_skipped", …)` per D-999.3-OBSERVABILITY.
3. Return `{ executed: false, failed: false }` — same as today.

This gives a single source of truth for "skip a stale assign action" and adds
the structured observability the design doc wants.

**Confidence: HIGH** — verified directly. (`src/dispatch/assign-executor.ts:38-71,362-365`)

### V6. `task.frontmatter.lastTransitionAt` source of truth (answered)

**Finding:** `transitionTask` at `src/store/task-mutations.ts:171` sets
`task.frontmatter.lastTransitionAt = now` on every non-no-op transition. The
`lastTransitionAt` field is non-optional in the Zod schema at
`src/schemas/task.ts:153`:
```ts
lastTransitionAt: z.string().datetime().describe("Last status change timestamp"),
```
And it's stamped on `create` at `task-store.ts:389` (initial value = createdAt).

**Caveat:** Idempotent same-status transitions short-circuit at
`task-mutations.ts:158-160` and DO NOT update `lastTransitionAt`:
```ts
if (currentStatus === newStatus) {
  return task;
}
```
So `lastTransitionAt` is the ISO timestamp of the most recent *real* status
change. This is the desired semantic — `expected.lastTransitionAtBefore` is
"transitioned since I queued this action," and idempotent transitions don't
count.

**Other fields that could mutate without bumping `lastTransitionAt`:** lease
acquisition (writes via `store.save` in `lease.ts`), metadata patches (via
`store.update`), body updates (via `store.updateBody`), `addDep`/`removeDep`.
None of these change `frontmatter.status` so they shouldn't matter for the
precondition check (which compares `task.frontmatter.lastTransitionAt`
against `expected.lastTransitionAtBefore` AFTER also confirming `status` and
`leaseAgent`).

**One subtle edge case:** `assign-executor.ts:107` does `await store.save(leasedTask)`
to write the correlationId into metadata after `acquireLease`. This is a write
to the task file but NOT a status transition. `lastTransitionAt` is unchanged
by this write. Good — that's the desired behavior. The status change happened
inside `acquireLease` (via `store.transition`), which does bump it.

**Confidence: HIGH** — schema, mutation logic, and idempotency early-return all
inspected. (`src/schemas/task.ts:153`, `src/store/task-mutations.ts:158-171`,
`src/store/task-store.ts:389`).

## Implementation Sketch (per handler)

### New module: `src/dispatch/precondition.ts`

```ts
import type { Task, TaskStatus } from "../schemas/task.js";
import type { SchedulerAction } from "./types.js";

export interface PreconditionActual {
  status: TaskStatus;
  leaseAgent: string | null;
  lastTransitionAt: string;
}

export type MatchResult =
  | { ok: true }
  | { ok: false; reason: "status_mismatch" | "leaseAgent_mismatch" | "lastTransitionAt_advanced"; actual: PreconditionActual };

export function matchesExpected(
  task: Task,
  expected: SchedulerAction["expected"] | undefined,
): MatchResult {
  if (!expected) return { ok: true };
  const actual: PreconditionActual = {
    status: task.frontmatter.status,
    leaseAgent: task.frontmatter.lease?.agent ?? null,
    lastTransitionAt: task.frontmatter.lastTransitionAt,
  };
  if (expected.status !== undefined && expected.status !== actual.status) {
    return { ok: false, reason: "status_mismatch", actual };
  }
  if (expected.leaseAgent !== undefined && expected.leaseAgent !== actual.leaseAgent) {
    return { ok: false, reason: "leaseAgent_mismatch", actual };
  }
  if (
    expected.lastTransitionAtBefore !== undefined &&
    actual.lastTransitionAt > expected.lastTransitionAtBefore
  ) {
    return { ok: false, reason: "lastTransitionAt_advanced", actual };
  }
  return { ok: true };
}
```

Co-located unit tests at `src/dispatch/__tests__/precondition.test.ts`. Pure
function — fastest possible test surface.

## Risks / Gotchas

### R1. Double-wrapping `lockManager` on `handleAssign` causes deadlock

**Setup:** `executeAssignAction` already wraps its body with
`config.lockManager.withLock(action.taskId, …)` at lines 362-365. If the planner
adds another `config.lockManager.withLock` at the `handleAssign` boundary
(`lifecycle-handlers.ts:187`), the inner queues forever waiting on itself.
**Mitigation:** Per V1/V3, place the precondition check inside
`executeAssignAction`, NOT at the `handleAssign` boundary. Plan must call this
out explicitly. Pre-merge sanity test: outer wrap + inner wrap on the SAME
lockManager (write a deliberate test that demonstrates the deadlock is real,
or a positive test that confirms our chosen design has only one wrap).

### R2. Schema migration required

**Setup:** `EventType` is a closed Zod enum (`src/schemas/event.ts:17-160`).
Calling `logger.log("scheduler_action_skipped", …)` without first adding the
new value will fail Zod validation at log time, NOT at compile time (the type
parameter is `EventType` which is `z.infer<typeof EventType>`, but Zod's enum
emits a unioned string type — so TS won't catch the missing value, runtime
will). **Mitigation:** Add the schema entry as the FIRST commit in the phase,
before any handler change. This is also a downstream consumer concern — any
event consumer that does its own enum validation (events MCP, dashboards) must
be inspected.

### R3. CLAUDE.md "Dispatch chain is fragile" warning

**Setup:** CLAUDE.md explicitly flags the dispatch chain
(`scheduler.ts → task-dispatcher.ts → action-executor.ts → assign-executor.ts`)
as "tightly coupled. Changes cascade." This phase touches all four files.
**Mitigation:** Run the full integration suite after each handler change, not
just unit tests. Particularly watch `dag-scheduler-integration.test.ts`,
`completion-enforcement.test.ts`, `dag-completion-enforcement.test.ts`,
`callback-integration.test.ts` — any of these could trip on a new skip event
appearing where the existing assertion didn't expect one.

### R4. Existing `assign-executor.ts` already has TWO precondition-like checks

**Setup:** `assign-executor.ts:68-71` checks `if (latest.frontmatter.status !== "ready")` and `:73-77` checks `if (isLeaseActive(latest.frontmatter.lease))`. Both early-return without emitting an event today. The new `matchesExpected` would supersede these for actions that carry `expected`. **Mitigation:** Decide explicitly:
- Option A (recommended): Replace both checks with `matchesExpected` when `action.expected` is set, fall through to legacy checks when it isn't (backwards-compat for action-producers that don't populate `expected`).
- Option B: Keep both legacy checks as belt-and-suspenders, add `matchesExpected` on top.

D-999.3-BACKWARDS-COMPAT favors Option A's spirit (the new check supersedes the old one for actions that carry `expected`); the planner should call this out and document in `assign-executor.ts` why both paths exist (for tests/external producers without `expected`).

### R5. `expected.lastTransitionAtBefore` boundary semantics

**Setup:** CONTEXT.md D-999.3-MATCH-FN says "`task.frontmatter.lastTransitionAt
> expected.lastTransitionAtBefore` → mismatch (task transitioned since action
was queued). Equality is treated as 'no mismatch' — same instant means the
action is acting on the same state it was queued for." ISO timestamps with
millisecond precision should make exact-equality between
`task.frontmatter.lastTransitionAt` (read by the action-builder) and the
later-read `task.frontmatter.lastTransitionAt` (in the handler) the COMMON
case (the task didn't transition between scan and execute). String comparison
on ISO-8601 datetimes is correct ordering as long as both values use the same
format — verify with a quick test.

**Mitigation:** Unit-test L2 must include an exact-equality case asserting
`{ok: true}`, and a 1ms-later case asserting `lastTransitionAt_advanced`.

### R6. `lockManager` is shared between protocol layer and scheduler

**Setup:** `aof-service.ts:114` constructs ONE `InMemoryTaskLockManager` and
passes it to BOTH `protocolRouter` and `schedulerConfig`. So any taskId
acquired by the protocol path (e.g. handling an MCP `aof_task_complete`) blocks
a scheduler action wrap on the same task, and vice versa. This is by design
(BUG-04) and unchanged by this phase, but it means: if a scheduler-action
precondition wrap holds the lock, and an MCP message tries to mutate the same
task, the MCP path blocks until the wrap exits. Document the implication in
the plan: "the wrap must be brief — read, check, mutate, return — no long-running
work inside."

The bodies described above ARE brief: a `store.get`, a synchronous
`matchesExpected`, optionally a `store.transition`. Total work is bounded by
local FS I/O. Acceptable.

### R7. Test mock for `InMemoryTaskLockManager` is identity-pass

**Setup:** Existing tests (`lifecycle-handlers.test.ts:142-154`) use
`{ withLock: vi.fn(async (_id, fn) => fn()) }` as the mock. This mock does
NOT serialize. **Implication:** A unit test that wants to PROVE the wrap is
in place can only assert `expect(withLock).toHaveBeenCalledWith(taskId, fn)`,
not assert any interleaving behavior. For interleaving / re-entrancy assertions,
use the real `InMemoryTaskLockManager` (cheap to construct, no I/O). L8 must
use the real implementation.

### R8. The `assign-helpers.ts:111` "other side" of the race is NOT touched

**Setup:** Per CONTEXT.md and the design doc, `assign-helpers.handleRunComplete`
is explicitly out of scope. After this phase ships, the recovery-side
(`handleStaleHeartbeat`) will skip when its premise is stale; the assign-side
(`assign-helpers.ts:111`) continues to transition unconditionally based on its
own snapshot of the task at line 107 (`store.get`). Fix A's mutex contains the
filesystem outcome. **Implication for verification:** the L3 incident
regression test should NOT also exercise `assign-helpers.handleRunComplete` —
that's a separate phase. The test should set up the `blocked` state directly
(via `store.transition`) and then fire `stale_heartbeat`, observing the skip.
Including assign-helpers in the test would conflate two phases.

## Open Questions

1. **Does any external action-producer set `SchedulerAction.fromStatus` independently of the scheduler?**
   - What we know: `fromStatus` is set only by `checkExpiredLeases` and `checkBacklogPromotion` per CONTEXT.md and confirmed by grep.
   - What's unclear: whether a future external producer might construct a `SchedulerAction` directly. Today's switch in `action-executor.ts:55` is the only consumer — no external producers exist.
   - Recommendation: treat `SchedulerAction` as a private dispatch-internal type. Document this in a comment on the interface.

2. **Should the `task_gone` skip path emit `scheduler_action_skipped`, or stay silent (existing behavior)?**
   - What we know: Today, `handleStaleHeartbeat:33-36` and the analogous early-returns in other handlers log `log.warn("task not found, skipping")` but emit no structured event.
   - What's unclear: Whether retroactively adding `scheduler_action_skipped` for the `task_gone` reason changes any consumer's expectations.
   - Recommendation: emit it for symmetry — it makes the events.jsonl record self-describing ("a queued action was skipped, here's why"). All four reasons (`task_gone`, `status_mismatch`, `leaseAgent_mismatch`, `lastTransitionAt_advanced`) get the same event shape. Document in plan.

3. **For `expected.leaseAgent` set but `task.frontmatter.lease === undefined` — mismatch reason `leaseAgent_mismatch` is a slight misnomer (the lease was cleared, not changed).**
   - What we know: D-999.3-MATCH-FN says "If `expected.leaseAgent` is set but `task.frontmatter.lease` is undefined → mismatch (lease was cleared)."
   - What's unclear: whether to emit a more specific reason like `lease_cleared`.
   - Recommendation: keep one reason `leaseAgent_mismatch` for both cases (different agent, no agent). The `actual.leaseAgent: null` field in the payload distinguishes them. Simpler enum, same diagnostic value.
