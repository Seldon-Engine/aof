<rescope_note>
## Rescope Note (2026-04-25)

This phase originally locked a 7-plan / 4-wave scope around a `SchedulerAction.expected` precondition envelope spanning all five state-mutating handlers (`handleStaleHeartbeat`, `handleExpireLease`, `handleAssign`, `handleSlaViolation`, `handlePromote`) plus a new `precondition.ts` helper, a new `scheduler_action_skipped` Zod event type, and scheduler-side population at every action-construction site. ~120 LOC prod + ~250 LOC test.

User pushback: "we are introducing a shitload of complexity, going in the wrong direction." Re-evaluation:

- **Production has shown the race ONCE, in ONE handler** (`handleStaleHeartbeat`, 2026-04-15).
- The other four handlers are theoretical race candidates — no incident.
- The existing fix (Fix A, v1.14.8 per-task transition mutex) already serializes the rename mechanically, so even if the race fires, the filesystem outcome is consistent. The remaining problem is wasted-action observability + not running stale recovery code on a task that already moved on.
- The minimum fix is ~10 LOC at the top of `handleStaleHeartbeat`: re-read status, skip if no longer `in-progress`. Plus an optional second guard for lease-agent re-assignment.

Decision: scope down to the narrow Phase 1 patch only. Re-open the broader envelope generalization if and when production evidence shows the same race shape in another handler. The rejected 7-plan scope is recoverable from git history (commits up to `2e6cc06`).
</rescope_note>

<domain>
## Phase Boundary

Add two precondition guards at the top of `handleStaleHeartbeat` (`src/dispatch/recovery-handlers.ts:26`) so the function no-ops when its dispatch-time premise no longer holds:

1. **Status guard** — if `staleTask.frontmatter.status !== "in-progress"`, log + return `{ executed: false, failed: false }`. Covers the primary 2026-04-15 incident shape: `assign-helpers.ts:111` (in-process completion enforcement) wins the race and transitions the task to `blocked`/`done`/`failed` first; the scheduler's stale-heartbeat action then fires and would otherwise re-transition the task to `ready` (or run another full recovery pass).
2. **Lease-agent guard** — if `action.agent` is set AND `staleTask.frontmatter.lease?.agent !== action.agent`, log + return the same skip result. Covers the re-dispatch shape: action queued for agent X but the task got re-leased to agent Y between queue-time and execute-time.

That is the entirety of in-scope source change: ~17 LOC in one file. Plus one new bug-NNN regression test (`bug-2026-04-15-stale-heartbeat-precondition.test.ts`) using `createTestHarness`, plus two unit tests added to the existing `recovery-handlers.test.ts`.

**Out of scope (deferred — see Rescope Note above for the original locked scope, archived in git up to commit `2e6cc06`):**
- `SchedulerAction.expected?: { status?, leaseAgent?, lastTransitionAtBefore? }` envelope on the type.
- New `src/dispatch/precondition.ts` helper module + `matchesExpected` + `logSkippedSafely`.
- New `scheduler_action_skipped` value in the `EventType` Zod enum.
- Scheduler-side population at action-construction sites (`scheduler.ts:110`, `:188`, `scheduler-helpers.ts:60`, `:108`, `task-dispatcher.ts`).
- Precondition guards on `handleExpireLease`, `handleAssign`/`executeAssignAction`, `handleSlaViolation`, `handlePromote`.
- Any `lockManager.withLock` wrapping (handleStaleHeartbeat does not currently use it; the narrow guard runs as a read-only pre-check followed by Fix A's mutex-protected `store.transition` — sufficient for this race shape).
- Refactoring `assign-helpers.ts:111`. That path is NOT a SchedulerAction handler; envelope/guards don't apply. Fix A serializes the rename; the new recovery-side guard skips when assign-helpers wins.

Re-open trigger: production evidence (events.jsonl + duplicate-file pattern in `~/.aof/data/tasks/<status>/`) of the same race shape in `handleExpireLease`, `handleAssign`, `handleSlaViolation`, or `handlePromote`. Until then: YAGNI.

<decisions>
## Implementation Decisions (LOCKED)

- **D-999.3-NARROW-SCOPE:** Single-file, single-handler patch in `src/dispatch/recovery-handlers.ts`. No new modules, no new schema values, no new helpers, no new event types. The fix matches what was needed in production, no more.
- **D-999.3-PRECONDITION-GUARD:** Two guards, in this order: status check (`!== "in-progress"`), then lease-agent check (`action.agent && lease?.agent !== action.agent`). Both placed immediately after the existing `if (!staleTask)` early-return at line ~37, before the `staleCorrelationId` read. Both return `{ executed: false, failed: false }` — the action was queued but did not fire because its premise no longer holds. Neither success nor failure.
- **D-999.3-OBSERVABILITY-VIA-LOG:** Skip events emit a single `log.info(...)` via the existing `recovery-handlers` pino logger — NO new EventLogger event type. The 2026-04-15 incident was diagnosable from log lines + duplicate-file presence; structured event-log telemetry is unnecessary for a race we expect to be rare. If skip frequency turns out to matter for ops, add the event then.
- **D-999.3-LEASEAGENT-NULLABLE-COMPARE:** Lease-agent guard fires when `action.agent` is truthy AND `staleTask.frontmatter.lease?.agent` differs (including the `lease cleared` case where the optional chain returns `undefined`). When `action.agent` is undefined (older queueing paths), the lease-agent guard short-circuits to "no check" and the function continues. Backwards-compatible.
- **D-999.3-RACE-WINDOW-TRADEOFF:** A check-then-act window remains between the precondition guard and the eventual `store.transition` call further down the function body. Closing it fully would require either wrapping the body in `config.lockManager.withLock` (doesn't help — assign-helpers does not go through that lock) or pushing the precondition into store.transition itself (broader contract change). We accept the residual sub-millisecond window because Fix A's mutex inside `store.transition` (v1.14.8) already prevents the duplicate-rename outcome even if the race fires. The narrow guard closes the original 9ms queue→execute window almost entirely.
- **D-999.3-RELEASE:** Patch release. Single-file additive change. No public API surface change. Backwards-compatible by construction. Ships as v1.18.0 patch.

<deferred>
## Deferred Ideas

(Recoverable from git history up to commit `2e6cc06` — the abandoned 7-plan scope.)

- **`SchedulerAction.expected` envelope across all five state-mutating handlers.** Re-open trigger: production evidence of the same race shape in another handler. Cost ~120 LOC prod + ~250 LOC test, vs the narrow ~17 LOC + ~80 LOC test. Not justified by current evidence.
- **`scheduler_action_skipped` Zod event type for structured telemetry.** Re-open trigger: ops needs to dashboard skip frequency. Until then, `log.info` lines suffice.
- **`config.lockManager.withLock` wrap on handleStaleHeartbeat body.** Doesn't actually close the residual race window — assign-helpers does not go through `config.lockManager`. Would only matter for a different race shape (handler-vs-handler concurrency on the same lockManager).
- **Action-level idempotency tokens (Phase 3 of design doc).** Premature without persistent action queues.
- **Multi-process locking** — out of scope per design doc Non-goals; AOF is single-process inside the OpenClaw gateway today.
- **Refactoring `assign-helpers.ts:111` to flow through SchedulerAction.** Would unify the two sides of the 2026-04-15 race. Held for a separate design phase — touches Phase 25 completion-enforcement contract.
</deferred>
