# Phase 999.3: Scheduler action pre-condition envelope — session-end handler dedupe — Context

**Gathered:** 2026-04-25
**Status:** Ready for planning
**Source:** Direct authoring from design doc (`.planning/fix-c-scheduler-ownership.md`, commit 693379a) + Fix A context (commit 746aee7 / v1.14.8) + post-Phase-46 codebase state. User direction: "one-shot this carefully" — gray areas resolved by Claude using strongest defaults from the design doc.

<domain>
## Phase Boundary

Add an `expected` pre-condition envelope to `SchedulerAction` so the four state-mutating scheduler handlers — `handleStaleHeartbeat`, `handleExpireLease`, `handleAssign`, and `handleSlaViolation`/`handlePromote` — re-verify their dispatch-time assumptions on entry **under the per-task lock** and no-op (with a structured skip log + EventLogger event) if the world has moved between dispatch and execution.

Eliminates the design root cause of the 2026-04-15 incident (TASK-2026-04-15-010) where two handlers responded to the same underlying session-end signal: `assign-helpers.ts` (in-process completion enforcement) and `handleStaleHeartbeat` (scheduler poll) both observed the task at `in-progress` and independently called `store.transition(...)` within the same event-loop tick. Fix A (per-task transition mutex inside `store.transition`, shipped v1.14.8) serializes the rename mechanically; Fix C — this phase — stops the second handler from acting at all when its premise is stale.

This phase covers BOTH shipping increments from the design doc (Phase 1 narrow `handleStaleHeartbeat` patch + Phase 2 general envelope across all four handlers) bundled into one GSD phase. Original design-doc release targets (v1.14.9 / v1.15.0) are stale — we are at v1.17.1; this ships as **v1.18.0 minor**.

**Out of scope (deferred / Non-goals):**
- Phase 3 of design doc — action-level idempotency tokens (`actionId = hash(type, taskId, lastTransitionAt)` + per-task ring buffer). Premature without persistent action queues.
- Multi-process task store safety (filesystem advisory locks, leases-via-rename). AOF is intra-process today; out of scope per design doc Non-goals.
- Event-log driven dedupe. Couples scheduling to event-log durability; out of scope per design doc Non-goals.
- Refactoring the in-process completion-enforcement path at `src/dispatch/assign-helpers.ts:111`. That path is NOT a `SchedulerAction` handler — it transitions directly via `store.transition`. The envelope doesn't apply; Fix A's mutex already serializes the rename, and after Fix C the recovery-side handler will skip with `expected.status` mismatch when assign-helpers wins the race.

</domain>

<decisions>
## Implementation Decisions (LOCKED)

### Scope cadence

- **D-999.3-SCOPE:** Bundle Phase 1 (narrow `handleStaleHeartbeat`-only patch) + Phase 2 (general envelope on all four handlers) into a single GSD phase shipping as v1.18.0 minor. Original v1.14.9 / v1.15.0 split-shipping plan from the design doc is obsolete (we are at v1.17.1). Splitting now adds workflow overhead without benefit; the narrow Phase 1 patch was meant to ship *before* Phase 2 to get the regression fix to users sooner — that ordering doesn't apply anymore.
- **D-999.3-RELEASE:** v1.18.0 minor. The `SchedulerAction` shape change is technically backwards-compatible (additive optional fields), but the contract change between scheduler-side action-builders and handler-side consumers warrants a minor bump per project convention.

### Envelope shape

- **D-999.3-ENVELOPE:** Add `expected?: { status?: TaskStatus; leaseAgent?: string; lastTransitionAtBefore?: string }` to `SchedulerAction` in `src/dispatch/types.ts`. All three sub-fields optional. All present in the design doc — no narrowing.
  - `status`: covers the primary 2026-04-15 incident shape (handler thought `in-progress`; another path moved it to `blocked`/`ready` first).
  - `leaseAgent`: covers re-dispatch shape (action queued for agent X but task got reassigned to agent Y).
  - `lastTransitionAtBefore`: covers the rare double-transition shape where status oscillates back to its original value between queue and execution (status check passes but the world moved twice). Compared with `>` against the task's current `frontmatter.lastTransitionAt`.
- **D-999.3-BACKWARDS-COMPAT:** `expected` is fully optional. A `SchedulerAction` without `expected` preserves today's behavior — handler proceeds with no precondition check. This keeps tests and any external action-producers compatible without forced migration. Phase 2's correctness gate is "all four handlers in scope populate via the scheduler", not "no SchedulerAction without expected anywhere".
- **D-999.3-fromStatus-MIGRATION:** The pre-existing `SchedulerAction.fromStatus` field (populated today by `checkExpiredLeases` and `checkBacklogPromotion`) is **kept** for now — its existing consumer (`logger.logTransition` in handlers) is orthogonal to the new precondition machinery. Action-builders that already set `fromStatus` ALSO set `expected.status` to the same value (eliminating the duplication is a deferred refactor; do not bundle).

### Handlers in scope (5 entries, 4 handler files)

| Handler | File | `expected` fields populated |
|---|---|---|
| `handleStaleHeartbeat` | `src/dispatch/recovery-handlers.ts:26` | `status: "in-progress"`, `leaseAgent: heartbeat.agentId`, `lastTransitionAtBefore: task.frontmatter.lastTransitionAt` |
| `handleExpireLease` | `src/dispatch/lifecycle-handlers.ts:26` | `status: "in-progress" \| "blocked"` (the two states the handler covers), `leaseAgent: lease.agent`, `lastTransitionAtBefore` |
| `handleAssign` | `src/dispatch/lifecycle-handlers.ts:187` (delegates to `assign-executor.ts`) | `status: "ready"`, `lastTransitionAtBefore` (no `leaseAgent` — pre-assignment, no lease yet) |
| `handleSlaViolation` | `src/dispatch/alert-handlers.ts:61` | `status: <evaluated-against>` (whatever status the SLA was checked against), `lastTransitionAtBefore` |
| `handlePromote` | `src/dispatch/lifecycle-handlers.ts:125` | `status: "backlog"`, `lastTransitionAtBefore` |

The design doc's 4-handler table merges `handleSlaViolation`/`handlePromote` into one row; treating them as separate populations is clearer and matches the codebase split (`alert-handlers.ts` vs `lifecycle-handlers.ts`).

### Lock interaction (TOCTOU closure)

- **D-999.3-LOCK:** All five state-mutating handlers wrap their `(read → check → mutate)` body in `config.lockManager.withLock(action.taskId, body)` when `config.lockManager` is set. Standardizes the pattern `handleExpireLease` already uses (`lifecycle-handlers.ts:113`). Closes the TOCTOU window between precondition re-read and the eventual `store.transition` call.
- **Why not push the precondition into `store.transition` itself?** That would be the cleanest architecture (the mutex already lives there, per Fix A), but it (a) changes the public store interface, (b) couples the store layer to scheduler-action semantics, and (c) would require migrating all `store.transition` callers to a new shape. The handler-level wrap matches the existing `handleExpireLease` pattern exactly and keeps the store layer's contract pristine.
- **Graceful degradation:** When `config.lockManager` is undefined (test harnesses that opt out), handlers run their body without the wrap. Behavior matches today plus the new precondition check (which is still useful even without the lock — closes most of the race window, though not all).
- **Compatibility with Fix A:** Fix A's mutex inside `store.transition` is per-task and re-entrant-safe via the same `lockManager` instance (single AOFService → single shared `lockManager`). Wrapping the handler body in `withLock(taskId, ...)` and then calling `store.transition` inside that body is fine — the inner `transition` call's own lock acquisition is a no-op when the outer lock is held by the same caller. Verify in plan: `InMemoryTaskLockManager.withLock` re-entrancy semantics.

### Mismatch behavior (skip + observability)

- **D-999.3-MISMATCH-RETURN:** Handlers that detect a stale precondition return `{ executed: false, failed: false }`. The action was queued but didn't fire because its premise no longer holds — neither success nor failure. Matches design doc and the existing `staleTask null` early-return pattern in `recovery-handlers.ts:35`.
- **D-999.3-OBSERVABILITY:** On every mismatch, emit BOTH:
  1. `log.info({ taskId, op, expected, actual, reason: "precondition_stale" }, "scheduler action skipped — precondition no longer holds")` — for human ops on the structured logger.
  2. `await logger.log("scheduler_action_skipped", "scheduler", { taskId, payload: { handler, expected, actual, reason } })` — structured EventLogger event for telemetry / future dashboards. Mirrors the existing `scheduler_action_failed` precedent at `src/dispatch/action-executor.ts:113`.
- **Event payload shape:** `{ handler: "handleStaleHeartbeat" | "handleExpireLease" | "handleAssign" | "handleSlaViolation" | "handlePromote", expected: { status?, leaseAgent?, lastTransitionAtBefore? }, actual: { status, leaseAgent: lease?.agent ?? null, lastTransitionAt }, reason: "status_mismatch" | "leaseAgent_mismatch" | "lastTransitionAt_advanced" | "task_gone" }`.
- **Why both?** Just-log-info would have made the 2026-04-15 incident harder to diagnose retroactively; just-event would lose the human-grep affordance during interactive debugging. Matches Phase 46's telemetry-heavy rollout posture.

### Scheduler-side population (where the envelope is built)

- **D-999.3-SCHEDULER-POPULATION:** Each action-construction site populates `expected` based on what it observed at queue time. Sites:
  - `checkExpiredLeases` (`src/dispatch/scheduler-helpers.ts:60`) — already sets `fromStatus`; adds `expected.status = task.frontmatter.status`, `expected.leaseAgent = lease.agent`, `expected.lastTransitionAtBefore = task.frontmatter.lastTransitionAt`.
  - `checkBacklogPromotion` (`src/dispatch/scheduler-helpers.ts:108`) — adds `expected.status = "backlog"`, `expected.lastTransitionAtBefore`.
  - Stale-heartbeat action site (`src/dispatch/scheduler.ts:110` inside `runPoll`) — adds `expected.status = "in-progress"`, `expected.leaseAgent = heartbeat.agentId`, `expected.lastTransitionAtBefore = task.frontmatter.lastTransitionAt`.
  - SLA violation site (`src/dispatch/scheduler.ts:188`) — adds `expected.status` matching the status the SLA evaluator examined, `expected.lastTransitionAtBefore`.
  - Hop-timeout site (`src/dispatch/scheduler.ts:202` → `checkHopTimeouts`) — populate matching the hop's evaluated status. (Hop-timeout actions today produce escalation alerts; verify whether they hit the same handler shape.)
  - `handleAssign` candidate-selection site (`task-dispatcher.ts`) — `expected.status = "ready"`, `expected.lastTransitionAtBefore`.
- **Single helper or per-site population?** Per-site. Each site already has the relevant task variable in scope; a single helper would force re-fetch or awkward parameter passing. Per-site is ~3 lines per site, ~5 sites = ~15 lines.

### Precondition evaluation

- **D-999.3-MATCH-FN:** Add a small pure helper `matchesExpected(task: Task, expected: SchedulerAction["expected"]): { ok: true } | { ok: false; reason: string; actual: {...} }` co-located in `src/dispatch/types.ts` or a new `src/dispatch/precondition.ts`. Used by all five handlers. Returns the structured `actual` snapshot for inclusion in the skip log + event payload.
- **Comparison rules:**
  - `expected.status`: strict equality with `task.frontmatter.status`.
  - `expected.leaseAgent`: strict equality with `task.frontmatter.lease?.agent`. If `expected.leaseAgent` is set but `task.frontmatter.lease` is undefined → mismatch (lease was cleared).
  - `expected.lastTransitionAtBefore`: `task.frontmatter.lastTransitionAt > expected.lastTransitionAtBefore` → mismatch (task transitioned since action was queued). Equality is treated as "no mismatch" — same instant means the action is acting on the same state it was queued for.
- **Order of checks:** task-existence → status → leaseAgent → lastTransitionAtBefore. Short-circuit on first mismatch. Reason in skip event reflects the first mismatch found.

### Test strategy

- **D-999.3-TEST-MATRIX:** "One-shot carefully" — go thorough.
  1. **Incident-shape regression** — `src/dispatch/__tests__/bug-2026-04-15-stale-heartbeat-precondition.test.ts` (follows the project's `bug-NNN-description.test.ts` regression naming). Reproduces the exact 2026-04-15 race: task is in-progress, another path transitions it to `blocked`, `handleStaleHeartbeat` action fires afterward. Asserts: `actionsFailed === 0`, `actionsExecuted === 0`, `scheduler_action_skipped` event present in events.jsonl with `expected.status === "in-progress"` and `actual.status === "blocked"`. Required by design doc success criterion #2.
  2. **Per-handler precondition tests** — for each of the five handlers, a test per precondition dimension (`status`, `leaseAgent`, `lastTransitionAtBefore`). ~15 tests. Each test sets up the dispatch-time state, mutates the task to violate one precondition, dispatches the action, asserts skip + structured event.
  3. **Negative tests** — for each handler, one test where the precondition holds end-to-end. Asserts the handler proceeds normally and emits its existing transition event (no `scheduler_action_skipped`). Guards against regressions in the happy path.
  4. **Lock interaction test** — verify `lockManager.withLock` re-entrancy: outer `withLock` in handler + inner `withLock` inside `store.transition` (Fix A) does not deadlock.
  5. **Backwards-compat test** — `SchedulerAction` without `expected` set: handler proceeds without precondition check, behavior matches today. Guards graceful-degradation invariant.
  6. **Fix A regression** — `src/store/__tests__/task-store-concurrent-transition.test.ts` MUST continue passing unchanged. Fix C does not weaken Fix A's invariant.
  7. **Integration sweep** — no unresolved `scheduler_action_failed` of type `stale_heartbeat` in any integration test's events.jsonl (success criterion #3).

### Claude's Discretion

- Choice between co-locating `matchesExpected` in `types.ts` vs new `precondition.ts` file — both fine. Default: new `src/dispatch/precondition.ts` (single-purpose, easier to test in isolation).
- Exact wording of `log.info` skip messages — keep consistent with existing `recovery-handlers.ts:34` and `lifecycle-handlers.ts:73` style.
- Whether to deduplicate the `fromStatus` / `expected.status` fields in a follow-up phase — defer; out of scope here.
- Whether `handleExpireLease`'s status-pair check (`"in-progress" | "blocked"`) is expressed as `expected.status` (single value match) or a tagged variant — default: populate `expected.status` with the actual observed status (one of the two), not a union, so the precondition check stays simple equality.

- **Handler signature**: `(action, store, logger, ...) => Promise<ActionHandlerResult>` — uniform across `lifecycle-handlers.ts`, `recovery-handlers.ts`, `alert-handlers.ts`. Add the precondition wrap at the top of each function body, not via decorator.
- **Event-logger try/catch best-effort**: every existing `logger.log(...)` and `logger.logTransition(...)` is wrapped in `try { ... } catch (err) { log.warn(...); }` — do the same for the new `scheduler_action_skipped` log.
- **Test harness**: `createTestHarness()` for integration, `createMockStore()`/`createMockLogger()` for unit. Existing handler tests in `src/dispatch/__tests__/{recovery,lifecycle,alert}-handlers.test.ts` provide the template for the new precondition tests.

### Integration Points
- `SchedulerConfig.lockManager` is plumbed through every dispatch entry point (plugin and standalone adapters) — no new config wiring needed.
- `SchedulerAction` is consumed only by `action-executor.ts` (`switch (action.type)`); no external consumers to migrate.

<specifics>
## Specific Ideas

- **Compatibility with Fix A's re-entrant lock semantics:** verify in plan that `InMemoryTaskLockManager.withLock(taskId, body)` is safe when `body` itself eventually calls `store.transition` (which acquires the same per-task lock from inside `task-store.ts`). If the lock is non-re-entrant, the inner acquisition would deadlock. Mitigation if non-re-entrant: have the handler call `store.save` / lower-level mutators that bypass the inner lock, or refactor `store.transition` to detect "already inside lock" via an option flag. Decide during research/planning, not now.
- **Scheduler-action source-of-truth for `lastTransitionAtBefore`:** read directly from `task.frontmatter.lastTransitionAt` at action-construction time. Don't depend on `lease.acquiredAt` or any other timestamp — `lastTransitionAt` is the canonical "when did this task last move" value.
- **Skip log output during normal operation:** in healthy steady-state, no actions should be skipped (preconditions hold). A baseline of skip events appearing during integration tests probably indicates a TEST bug, not production drift — call out in test review.

</specifics>

<deferred>
## Deferred Ideas

- **Phase 3 of design doc — action-level idempotency tokens.** Per-task ring buffer of recently-completed action IDs, derived from `(type, taskId, lastTransitionAt)`. Guards against scheduler-restart action retries. Per design doc Non-goals: premature today, valuable once persistent action queues exist. Re-evaluate if AOF ever moves to a persistent action queue (e.g. SQLite-backed).
- **`fromStatus` field deprecation / merge into `expected.status`.** The existing `SchedulerAction.fromStatus` (used by `logger.logTransition` callers) duplicates `expected.status` for action-builders that populate both. Tracking removal as a separate cleanup phase — bundling into 999.3 risks scope creep.
- **Multi-process locking** (filesystem advisory locks or leases-via-rename). Out of scope per design doc Non-goals. Re-evaluate if AOF ever spawns standalone daemon processes alongside the in-process plugin (post-Phase 43, daemon IS the single authority but only one daemon process per install today).
- **Refactoring `assign-helpers.ts:111` (in-process completion enforcement)** to flow through a `SchedulerAction` envelope. Would unify the two sides of the 2026-04-15 race under one mechanism. Held for a separate design phase — touches Phase 25 completion-enforcement contract.

</deferred>
