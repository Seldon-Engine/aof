## Phase Details

### Phase 4: Memory Fix & Test Stabilization
**Goal**: Memory subsystem works reliably — inserts never crash, search returns correct results, operators can diagnose and repair index health, and the full test suite passes green
**Depends on**: Nothing (standalone P0 fix, first phase of v1.1)
**Requirements**: MEM-01, MEM-02, MEM-03, MEM-04, MEM-05, MEM-06, CI-02
**Success Criteria** (what must be TRUE):
  1. Inserting 10,000+ memory chunks completes without crash — HNSW index resizes automatically when capacity is reached
  2. `aof memory health` outputs index count, SQLite count, fragmentation percentage, and last rebuild time
  3. `aof memory rebuild` reconstructs HNSW from SQLite with progress output, and search returns correct results afterward
  4. After a save/load cycle (daemon restart), memory search returns the same results as before the restart
  5. `npm test` passes with 0 failures (pre-existing test failures in runDaemonStep and OpenClawAdapter suites are fixed or properly skipped with tracking comments)
**Plans**: 3 plans

Plans:
- [x] 04-01-PLAN.md — HNSW index hardening, startup parity check, auto-rebuild, integration tests (MEM-01, MEM-02, MEM-03, MEM-04)
- [x] 04-02-PLAN.md — Fix pre-existing test failures across executor and lifecycle suites (CI-02)
- [x] 04-03-PLAN.md — CLI health and rebuild commands for memory index (MEM-05, MEM-06)

### Phase 5: CI Pipeline
**Goal**: Every PR and release is automatically validated and packaged — CI catches regressions on push, and releases produce downloadable artifacts with changelogs
**Depends on**: Phase 4 (test suite must be green before CI goes live)
**Requirements**: CI-01, CI-03, CI-04
**Success Criteria** (what must be TRUE):
  1. Pushing a commit or opening a PR to main triggers a GitHub Actions workflow that runs typecheck, build, and tests — and it passes green
  2. Triggering the release workflow creates a GitHub Release with a version-tagged tarball artifact attached and a conventional-commit changelog in the release body
  3. CHANGELOG.md in the repository is updated on each release with the new version's changes
**Plans**: 2 plans

Plans:
- [x] 05-01-PLAN.md — CI validation workflow (typecheck/build/test on PR/push) and release-it CHANGELOG.md config (CI-01, CI-04)
- [x] 05-02-PLAN.md — Release workflow (tag-triggered tarball build + upload) and build script (CI-03)

### Phase 6: Installer
**Goal**: A person who has never seen AOF can install it on a machine with Node >= 22 and OpenClaw using a single curl command — and running it again upgrades without data loss
**Depends on**: Phase 5 (installer downloads release tarballs that CI creates)
**Requirements**: INST-01, INST-02, INST-03, INST-04, INST-05, INST-06
**Success Criteria** (what must be TRUE):
  1. `curl -fsSL <url>/install.sh | sh` on a fresh machine with Node >= 22 and OpenClaw installs AOF, scaffolds directories, and wires it as an OpenClaw plugin — AOF commands work immediately afterward
  2. Running the installer on a machine without Node >= 22 or without OpenClaw prints a clear error message and exits without modifying anything
  3. Running the installer on an existing AOF installation upgrades the code without losing tasks, events, memory data, or configuration
  4. The installer downloads from the real GitHub repository URL (not a placeholder)
**Plans**: 2 plans

Plans:
- [x] 06-01-PLAN.md — Fix stubs and placeholders: GITHUB_REPO constant, extractTarball() implementation, package-lock.json in tarball (INST-02, INST-06)
- [x] 06-02-PLAN.md — Shell installer + Node.js setup command: prerequisite checks, download/extract, wizard scaffolding, migrations, OpenClaw plugin wiring (INST-01, INST-03, INST-04, INST-05)

### Phase 7: Projects (existing primitive — verification & gap-filling)
**Goal**: The existing project primitive works end-to-end — close identified gaps in tool scoping, dispatch filtering, and memory pool isolation so multiple projects run on the same AOF instance with complete isolation
**Depends on**: Phase 4 (working memory needed for pool isolation verification)
**Requirements**: PROJ-01, PROJ-02, PROJ-03, PROJ-04, PROJ-05, PROJ-06
**Success Criteria** (what must be TRUE):
  1. `aof project create --template` scaffolds a project directory with manifest, task directories, and memory config — and the project appears in `aof project list`
  2. Dispatching a task with a project ID routes it to the correct project's task directory, and the scheduler picks it up from that project-specific store
  3. An agent assigned to Project A cannot be dispatched tasks from Project B (dispatcher filters by project participants)
  4. Memory search within Project A returns zero results from Project B's memory pool, even when both pools contain semantically similar content
  5. An integration test exercises the full lifecycle: create project, create task, dispatch, verify task isolation, verify memory isolation
**Plans**: 3 plans

Plans:
- [x] 07-01-PLAN.md — ToolContext project propagation, project-scoped store resolution in adapter, participant filtering in dispatcher (PROJ-01, PROJ-02, PROJ-03)
- [x] 07-02-PLAN.md — Per-project memory isolation with separate SQLite DB and HNSW index per project (PROJ-04)
- [x] 07-03-PLAN.md — CLI commands, OpenClaw project tools, companion skill, end-to-end integration test (PROJ-05, PROJ-06)
