# Roadmap: AOF Production Readiness

## Overview

AOF's core architecture is solid -- scheduler, task store, protocols, and event logging all work and are well-tested. What remains is production hardening: making the poll loop and shutdown resilient (Foundation), giving the daemon a clean lifecycle under launchd supervision (Daemon), building the real gateway executor that has never existed beyond mocks (Gateway), adding automatic recovery for failed tasks and stuck sessions (Self-Healing), and making the install experience work for someone who is not the author (Install). Each phase unblocks the next: you cannot test daemon reliability with a poll loop that hangs, cannot integration-test gateway dispatch without a reliable daemon, cannot self-heal without a working executor, and install wraps the whole thing for end users.

## Phase Details

### Phase 1: Foundation Hardening
**Goal**: Scheduler can be stopped, started, and restarted without leaving orphaned state, hung promises, or lost tasks
**Depends on**: Nothing (first phase)
**Requirements**: FOUND-01, FOUND-02, FOUND-03, FOUND-04
**Success Criteria** (what must be TRUE):
  1. A scheduler poll that takes longer than the configured timeout is aborted and the next poll proceeds normally
  2. Stopping the daemon completes all in-flight task transitions before exiting (no half-written state on disk)
  3. After a hard kill (SIGKILL) and restart, all tasks that were in-progress are reclaimed and re-dispatched within 2 poll cycles
  4. A task that fails due to a rate limit is retried with backoff; a task that fails due to a missing agent is dead-lettered immediately

### Phase 2: Daemon Lifecycle
**Goal**: AOF runs as a launchd/systemd-supervised service with health reporting, clean start/stop, and automatic crash recovery via the OS
**Depends on**: Phase 1
**Requirements**: DAEM-01, DAEM-02, DAEM-03, DAEM-04, DAEM-05
**Success Criteria** (what must be TRUE):
  1. `aof daemon install` on macOS generates a valid launchd plist and the daemon starts under launchd supervision
  2. `aof daemon install` on Linux generates a valid systemd unit and the daemon starts under systemd supervision
  3. The health endpoint returns JSON with task counts, uptime, version, and component status before the PID file is written
  4. After killing the daemon process with SIGKILL, launchd/systemd restarts it and the health endpoint responds within 30 seconds
  5. `aof daemon stop` drains in-flight work and removes the PID file cleanly (no orphaned state)
**Plans**: 3 plans

### Phase 3: Gateway Integration
**Goal**: Tasks are dispatched to real agents via the OpenClaw gateway and tracked from spawn to completion
**Depends on**: Phase 2
**Requirements**: GATE-01, GATE-02, GATE-03, GATE-04, GATE-05
**Success Criteria** (what must be TRUE):
  1. A task promoted to ready is dispatched to a real agent session via the plugin-sdk and the task moves to done when the agent completes
  2. The adapter interface allows swapping OpenClaw for a different platform without changing dispatch logic
  3. Every dispatched session has a correlation ID linking the task, the agent session, and the completion event
  4. An agent session that produces no heartbeat for longer than the configured timeout is force-completed and the task is reclaimed
  5. An integration test suite runs dispatch-to-completion against a real (or simulated) gateway and passes in CI
**Plans**: 2 plans

Plans:

- [ ] 03-02-PLAN.md — Correlation IDs, adapter-aware session lifecycle, three mandatory integration test scenarios (GATE-03, GATE-04, GATE-05)

### Phase 4: Self-Healing
**Goal**: The system automatically recovers from transient failures without human intervention
**Depends on**: Phase 3
**Requirements**: HEAL-01, HEAL-02, HEAL-03
**Success Criteria** (what must be TRUE):
  1. A dead-lettered task is automatically resurrected after exponential backoff and re-dispatched (up to a configurable max attempts)
  2. A stuck session (agent has heartbeat but never completes) is detected and force-recovered after a configurable threshold
  3. When a provider hits repeated failures, its circuit breaker opens and tasks are not dispatched to it until the breaker resets -- other providers continue operating normally

### Phase 5: Install Experience
**Goal**: A new user can install, configure, and start AOF from scratch without reading source code or hitting path/config errors
**Depends on**: Phase 2 (daemon lifecycle must work for install to test)
**Requirements**: INST-01, INST-02, INST-03, INST-04
**Success Criteria** (what must be TRUE):
  1. Only one code path for plugin registration exists (the competing `integration.ts` path is removed)
  2. A config validation error produces a human-readable message with a fix suggestion (not raw Zod output)
  3. Running `aof init` from `/tmp` on a fresh user account completes without errors and produces a working configuration
  4. The interactive install wizard walks the user through gateway connection, org chart basics, and daemon setup with clear prompts
