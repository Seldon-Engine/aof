**Step 1: Extend AofMcpContext in src/mcp/shared.ts**

Add an optional `projectConfig` field to `AofMcpContext`:
```typescript
projectConfig?: ProjectManifest;
```
Import `ProjectManifest` from `../schemas/project.js`. Also update `createAofMcpContext` to load project config when projectId is provided -- reuse the project resolution that already happens there, read and parse the `project.yaml` manifest.

If projectId is NOT provided (legacy mode), leave projectConfig undefined.

**Step 2: Add workflow field to dispatchInputSchema in src/mcp/tools.ts**

Add a `workflow` field to `dispatchInputSchema` that accepts either:
- A string (template name to resolve)
- An object matching WorkflowDefinition schema (inline DAG)
- `false` (explicitly skip default workflow)

Use a Zod union: `z.union([z.string(), WorkflowDefinitionSchema, z.literal(false)]).optional()`

Import `WorkflowDefinition` schema from `../schemas/workflow-dag.js`.

**Step 3: Add workflow resolution logic in handleAofDispatch**

Before the `ctx.store.create()` call, add workflow resolution:

1. If `input.workflow` is a string:
   - Look up `ctx.projectConfig?.workflowTemplates?.[input.workflow]`
   - If not found, throw `McpError(ErrorCode.InvalidParams, "Unknown workflow template: {name}")`
   - If found, validate with `validateDAG()`, pass `{ definition, templateName: input.workflow }` to store.create

2. If `input.workflow` is an object (WorkflowDefinition):
   - Validate with `validateDAG(input.workflow)`
   - If validation errors, throw `McpError(ErrorCode.InvalidParams, "Invalid workflow DAG: {errors}")`
   - Pass `{ definition: input.workflow }` to store.create

3. If `input.workflow` is `false`:
   - Do not attach any workflow (explicit skip)

4. If `input.workflow` is undefined:
   - Do not attach any workflow (backward compatible -- agents are explicit, no auto-default per CONTEXT.md discretion note)

Pass the resolved workflow to `ctx.store.create()` via the existing `workflow` parameter.

**Step 4: Write tests in src/mcp/__tests__/dispatch-workflow.test.ts**

Follow existing test patterns in the codebase (vitest, mock store). Create a mock AofMcpContext with a projectConfig containing workflowTemplates. Test all 6 behaviors listed above. Use `handleAofDispatch` directly (unit test, not integration).

**Design decision (per CONTEXT.md Claude's Discretion):** No auto-default workflow when workflow param is omitted. Agents are explicit -- surprise workflow attachment would be confusing. This differs from CLI which has auto-default, but is appropriate for MCP tool usage where agents should opt-in to workflows.
