**In `src/cli/commands/memory.ts` — add `health` subcommand:**

1. **Add the command** within `registerMemoryCommands()`:
   ```typescript
   memory
     .command("health")
     .description("Show memory index health metrics")
     .option("--json", "Output as JSON", false)
     .action(async (opts: { json: boolean }) => { ... });
   ```

2. **Resolve paths** — Get `root` from `program.opts()["root"]`, derive:
   - `dbPath` = join(root, "memory.db") or from plugin config
   - `hnswPath` = dbPath.replace(/\.db$/, "-hnsw.dat")
   Check how existing memory commands resolve these paths (look at the `import` and `curate-generate` commands for the pattern).

3. **Open SQLite** — Use `initMemoryDb(dbPath, dimensions)` or raw `better-sqlite3` open. Get dimensions from the loaded HNSW index via `getNumDimensions()`.

4. **Load HNSW** — Create `new HnswIndex(dimensions)`, call `hnsw.load(hnswPath)`. If load fails (file missing), report as "HNSW index: NOT FOUND".

5. **Compute health metrics** per locked decisions:
   - `hnswCount`: `hnsw.count`
   - `sqliteCount`: `SELECT COUNT(*) as c FROM vec_chunks`
   - `syncStatus`: "ok" if counts match, "desynced" if not
   - `fragmentationPct`: `((hnsw.maxElements - hnsw.count) / hnsw.maxElements) * 100` (round to 1 decimal)
   - `lastRebuildTime`: `SELECT value FROM memory_meta WHERE key = 'last_rebuild_time'` (null if never rebuilt)
   - `pools`: `SELECT pool, COUNT(*) as count FROM chunks GROUP BY pool`

6. **Output format** — Per locked decisions:
   - Default: Human-readable table
   ```
   Memory Health Report
     HNSW count:      1234
     SQLite count:    1234
     Sync status:     ok
     Fragmentation:   15.3%
     Last rebuild:    2026-02-26T10:30:00

     Pool Breakdown:
       core           890
       agent-main     234
       agent-swe-ai   110
   ```
   - `--json`: `JSON.stringify(report, null, 2)`

7. **Clean up** — Close the SQLite database after the report.

**In `src/cli/__tests__/memory-health.test.ts`:**

Create unit tests for the health report generation logic. Extract the health computation into a pure function `computeHealthReport(db, hnsw)` that returns a typed report object, so it can be tested without running the full CLI.

Test cases:
- `returns ok sync status when counts match`
- `returns desynced status when counts differ`
- `computes fragmentation percentage correctly`
- `returns null for last rebuild time when never rebuilt`
- `includes per-pool breakdown`

**In `src/cli/commands/memory.ts` — add `rebuild` subcommand:**

1. **Add the command**:
   ```typescript
   memory
     .command("rebuild")
     .description("Rebuild HNSW index from SQLite")
     .option("--yes", "Skip confirmation prompt", false)
     .action(async (opts: { yes: boolean }) => { ... });
   ```

2. **Resolve paths** — Same pattern as health command.

3. **Open SQLite and count chunks**:
   ```typescript
   const totalChunks = (db.prepare("SELECT COUNT(*) as c FROM vec_chunks").get() as { c: number }).c;
   ```

4. **Check if daemon is running** — Look for PID file at `join(root, "daemon.pid")`. If exists, print warning:
   ```
   ⚠ Daemon appears to be running (PID file exists). The daemon will use the old index until restarted.
   ```
   Per research recommendation. Proceed anyway.

5. **Confirmation prompt** — Per locked decision: "Rebuild requires confirmation prompt: 'This will rebuild the HNSW index from SQLite (X chunks). Continue? [y/N]' — skippable with `--yes` flag."
   ```typescript
   if (!opts.yes) {
     const { confirm } = await import("@inquirer/prompts");
     const answer = await confirm({
       message: `This will rebuild the HNSW index from SQLite (${totalChunks} chunks). Continue?`,
       default: false,
     });
     if (!answer) { console.log("Aborted."); return; }
   }
   ```

6. **Record before stats** — Load existing HNSW (if exists) and record count before rebuild.

7. **Read all chunks from SQLite**:
   ```typescript
   const rows = db.prepare("SELECT chunk_id, embedding FROM vec_chunks").all();
   const chunks = rows.map(row => ({
     id: row.chunk_id,
     vector: Array.from(new Float32Array(row.embedding)),
   }));
   ```

8. **Progress bar** — Per locked decision: "shows a progress bar during rebuild (X/Y chunks indexed)":
   ```typescript
   import { SingleBar, Presets } from "cli-progress";

   const bar = process.stdout.isTTY
     ? new SingleBar({ format: "Indexing [{bar}] {percentage}% | {value}/{total} chunks | ETA: {eta}s" }, Presets.shades_classic)
     : null;

   bar?.start(chunks.length, 0);
   ```
   Per Pitfall 5 from research: check `process.stdout.isTTY` before using progress bars.

9. **Rebuild with progress** — Create new HnswIndex, build it chunk by chunk (or use `rebuild()` method if it accepts a progress callback). If `HnswIndex.rebuild()` doesn't support progress callbacks, do manual insertion:
   ```typescript
   const hnsw = new HnswIndex(dimensions);
   // Initialize with headroom capacity
   const capacity = Math.max(Math.ceil(chunks.length * 1.5), INITIAL_CAPACITY);
   // ... create index with capacity, then add chunks one by one with progress updates
   for (const chunk of chunks) {
     hnsw.add(chunk.id, chunk.vector);
     bar?.increment();
   }
   bar?.stop();
   ```
   If the HnswIndex class doesn't expose a way to do this (because `rebuild()` does it internally), either:
   - Modify `HnswIndex.rebuild()` to accept an `onProgress?: (done: number) => void` callback
   - Or construct the index manually from the CLI

10. **Save index** — `hnsw.save(hnswPath)`

11. **Update last rebuild time** in `memory_meta` table:
    ```typescript
    db.prepare("INSERT OR REPLACE INTO memory_meta (key, value) VALUES ('last_rebuild_time', datetime('now'))").run();
    ```

12. **Print summary table** — Per locked decision: "then a summary table with before/after stats at the end":
    ```
    Rebuild Complete
      Before:  0 elements (desynced)
      After:   1234 elements
      Duration: 2.3s
    ```

13. **Clean up** — Close SQLite database.

**Non-TTY fallback:** If `process.stdout.isTTY` is false, print simple line output instead of progress bar:
```
Indexing: 100/1234 chunks...
Indexing: 200/1234 chunks...
```
