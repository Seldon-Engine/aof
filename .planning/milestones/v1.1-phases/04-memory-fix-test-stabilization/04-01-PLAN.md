1. **Expose `maxElements` getter** — Add a getter that returns `this.index.getMaxElements()`. The health CLI (Plan 03) and fragmentation calculation need this.

2. **Expose `dimensions` getter** — Add a getter that returns `this.dimensions` for CLI index loading.

3. **Verify `ensureCapacity()` fires on all paths** — The existing code has `ensureCapacity()` called inside `add()`. Confirm this covers the post-load path (load → add should trigger resize if at capacity). If `ensureCapacity()` checks `count >= max`, this works. Verify the growth factor: currently `GROWTH_FACTOR = 2` (doubling). This is fine — research said Claude's discretion.

4. **Add capacity headroom in `rebuild()`** — Change `Math.max(chunks.length, INITIAL_CAPACITY)` to `Math.max(Math.ceil(chunks.length * 1.5), INITIAL_CAPACITY)`. This prevents the next insert after rebuild from immediately triggering a resize (Pitfall 4 from research).

5. **Log resize events** — When `ensureCapacity()` triggers a resize, log `memory.index.resized` event to JSONL AND print a debug-level line. Per locked decision: "log `memory.index.resized` event to JSONL AND print a debug-level line to daemon output (visible with -v)". The EventLogger access pattern: accept an optional `logEvent` callback in constructor or `add()` method, so VectorStore can wire it.

**In `src/memory/store/vector-store.ts`:**

1. **Add `hnswPath` to constructor** — The VectorStore needs to know where to save the index. Add `hnswPath?: string` parameter. Store as `private readonly hnswPath: string | null`.

2. **Add `saveIndex()` private method** — Calls `this.hnsw?.save(this.hnswPath)` if both exist. Per locked decision: "Index is persisted to disk after every write operation — crash safety over performance."

3. **Call `saveIndex()` after every mutation** — Add call after `insertChunk()`, `removeChunk()`, and `removeByFile()` — after the HNSW modification, outside the SQLite transaction.

4. **Add async mutex** — Create a simple promise-based `Mutex` class (15 lines, pattern from research). Wrap HNSW mutations (`insertChunk`, `removeChunk`, `removeByFile`) in mutex acquire/release. Per locked decision: "Concurrent memory writes from multiple agents are allowed, protected by a mutex around HNSW operations."

5. **Add `rebuilding` flag** — Add `private rebuilding = false` property. When `rebuilding` is true, `search()` should skip HNSW and fall through to sqlite-vec path. Per locked decision: "During rebuild, memory searches fall back to sqlite-vec path."

6. **Wire `hnswPath` from `registerMemoryModule()`** — Update the VectorStore constructor call in `src/memory/index.ts` to pass `hnswPath`.

**Anti-patterns to avoid:**
- Do NOT add incremental patching of HNSW — any desync triggers full rebuild (locked decision)
- Do NOT implement lazy loading of HNSW — index stays in memory permanently (locked decision)

**In `src/memory/index.ts` — modify `registerMemoryModule()`:**

1. **Add parity check after HNSW load** — After the existing `hnsw.load(hnswPath)` call (or after creating a fresh index), compare counts:
   ```typescript
   const hnswCount = hnsw.count;
   const sqliteCount = (db.prepare("SELECT COUNT(*) as c FROM vec_chunks").get() as { c: number }).c;
   ```
   Per locked decision: "On startup, run HNSW-SQLite parity check (count comparison) every time — negligible cost, catches drift early."

2. **Auto-rebuild on desync** — If `hnswCount !== sqliteCount`, call `rebuildHnswFromDb(db, hnsw)` and print a visible warning:
   ```
   [AOF] HNSW-SQLite desync detected (HNSW: ${hnswCount}, SQLite: ${sqliteCount}). Rebuilding index...
   ```
   Per locked decision: "If HNSW and SQLite are out of sync (any desync, regardless of magnitude): full rebuild from SQLite. SQLite is the source of truth. No incremental patching."

3. **Auto-rebuild on missing/corrupt index** — If the HNSW file doesn't exist or `hnsw.load()` throws, catch the error, log a warning, and rebuild from SQLite:
   ```
   [AOF] HNSW index missing or corrupt. Rebuilding from SQLite...
   ```
   Per locked decision: "If HNSW index is missing or corrupt: auto-rebuild from SQLite with a visible warning in daemon output (not silent, not fatal)."

4. **Update `rebuildHnswFromDb()`** — Ensure it uses 1.5x headroom capacity (matching the HnswIndex.rebuild change from Task 1). The function reads all chunks from SQLite and calls `hnsw.rebuild(chunks)`. Verify the existing code passes through to `HnswIndex.rebuild()` which now has headroom.

5. **Track last rebuild time** — After a rebuild (auto or manual), store the timestamp. Add a simple `memory_meta` table to SQLite (via schema.ts or inline):
   ```sql
   CREATE TABLE IF NOT EXISTS memory_meta (key TEXT PRIMARY KEY, value TEXT)
   ```
   After rebuild: `INSERT OR REPLACE INTO memory_meta (key, value) VALUES ('last_rebuild_time', datetime('now'))`.
   This is needed by `aof memory health` (Plan 03, MEM-05).

6. **Pass hnswPath to VectorStore** — Update the `new VectorStore(db, hnsw)` call to `new VectorStore(db, hnsw, hnswPath)` to enable save-after-write from Task 1.

**Anti-patterns to avoid:**
- Do NOT make the rebuild silent — always log a visible warning (locked decision)
- Do NOT do incremental patching — any desync = full rebuild (locked decision)
