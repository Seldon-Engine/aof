<task type="auto">
  <name>Task 1: Create CI validation workflow</name>
  <files>.github/workflows/ci.yml</files>
  <action>
Create `.github/workflows/ci.yml` with the following exact configuration:

**Triggers:**
- `on.push.branches: [main]`
- `on.pull_request.branches: [main]` with `types: [opened, synchronize, reopened, ready_for_review]`

**Concurrency:**
- `group: ci-${{ github.ref }}` with `cancel-in-progress: true`

**Job: `validate`**
- `name: Node ${{ matrix.node-version }}`
- `if: github.event.pull_request.draft != true` (skips draft PRs; evaluates true for push events since `github.event.pull_request` is null)
- `runs-on: ubuntu-latest`
- `strategy.fail-fast: false` (so Node 23 failure doesn't mask Node 22 success)
- `strategy.matrix.node-version: [22, 23]`

**Steps (in order):**
1. `actions/checkout@v6`
2. `actions/setup-node@v4` with `node-version: ${{ matrix.node-version }}` and `cache: 'npm'`
3. `npm ci` (Install dependencies)
4. `npm run typecheck` (Typecheck)
5. `npm run build` (Build)
6. `npm test` (Test)

Use `name:` labels on each step for readability. The workflow name should be `CI`.

Do NOT add `permissions`, `env`, or any other fields not listed above. Keep it minimal.

Ensure the `.github/workflows/` directory exists (it already does from `docs.yml` and `e2e-tests.yml`).
  </action>

<task type="auto">

  <files>.release-it.json</files>
  <action>
Update `.release-it.json` to enable CHANGELOG.md file generation.

In the `plugins.@release-it/conventional-changelog` section:
1. Change `"infile": false` to `"infile": "CHANGELOG.md"`
2. Add `"header": "# Changelog"` field (controls the file header)

Keep ALL other existing configuration unchanged -- the preset types, gitRawCommitsOpts, git config, github config, npm config, and hooks must remain exactly as they are.

The resulting plugin section should look like:
```json
"@release-it/conventional-changelog": {
  "preset": { ... },  // existing preset unchanged
  "infile": "CHANGELOG.md",
  "header": "# Changelog",
  "gitRawCommitsOpts": { ... }  // existing unchanged
}
```

This means when the user runs `npm run release` locally, release-it will:
1. Generate changelog from conventional commits
2. Write it to CHANGELOG.md (new behavior)
3. Include CHANGELOG.md in the release commit (automatic since it's a tracked file)
4. Push the commit + tag to origin
  </action>
