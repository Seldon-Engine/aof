  <name>Task 1: Create tarball build script</name>
  <files>scripts/build-tarball.mjs</files>
  <action>
Create `scripts/build-tarball.mjs` as an ESM Node.js script that assembles production files into a versioned tarball.

**Usage:** `node scripts/build-tarball.mjs v1.2.0`

**Implementation:**
1. Accept version string as first CLI argument (e.g., `v1.2.0`). Exit with error if missing.
2. Create a staging directory `.release-staging/` (use `mkdirSync` with `recursive: true`)
3. Copy production files into staging. The file list should mirror the `"files"` field in package.json plus `package.json` itself:
   - `dist/` (directory, recursive copy)
   - `prompts/` (directory, recursive copy)
   - `skills/` (directory, recursive copy)
   - `index.ts` (file)
   - `openclaw.plugin.json` (file)
   - `README.md` (file)
   - `package.json` (file)
   - `LICENSE` (file, optional -- use try/catch, skip if missing)
4. Use `cpSync(src, join(staging, src), { recursive: true })` for each item. For optional files, wrap in try/catch.
5. Create tarball: `tar -czf aof-{version}.tar.gz -C .release-staging .`
6. Clean up staging: `rm -rf .release-staging`
7. Print: `Created aof-{version}.tar.gz`

**Use only Node.js built-in modules:** `node:child_process` (execSync), `node:fs` (mkdirSync, cpSync, existsSync), `node:path` (join).

**Error handling:**
- If version argument is missing, print usage and `process.exit(1)`
- If any required file/directory doesn't exist (dist, prompts, skills, etc.), exit with clear error
- If tar command fails, the non-zero exit from execSync will propagate

Make the script executable (no shebang needed -- invoked via `node scripts/build-tarball.mjs`).

  <name>Task 2: Create release workflow</name>

  <action>
Create `.github/workflows/release.yml` with the following exact configuration:

**Workflow name:** `Release`

**Trigger:**
- `on.push.tags: ['v*']` (fires when release-it pushes a version tag)

**Concurrency:**
- `group: release-${{ github.ref }}` with `cancel-in-progress: false` (release runs must always complete)

**Permissions:**
- `contents: write` (needed to upload release assets)

**Job: `release`**
- `name: Build & Publish`
- `runs-on: ubuntu-latest`

**Steps (in order):**
1. `actions/checkout@v6`
2. `actions/setup-node@v4` with `node-version: 22` and `cache: 'npm'` (release builds on LTS only, not full matrix)
3. `npm ci` (Install dependencies)
4. `npm run typecheck` (Typecheck)
5. `npm run build` (Build)
6. `npm test` (Test -- safety net before publishing artifacts)
7. **Extract version** step:
   - `id: version`
   - `run: echo "version=${GITHUB_REF_NAME}" >> "$GITHUB_OUTPUT"`
8. **Build release tarball** step:
   - `run: node scripts/build-tarball.mjs ${{ steps.version.outputs.version }}`
9. **Upload tarball to GitHub Release** step:
   - `uses: softprops/action-gh-release@v2`
   - `with.files: aof-${{ steps.version.outputs.version }}.tar.gz`

Use `name:` labels on each step for readability.

Do NOT add Node version matrix (release builds on Node 22 only). Do NOT add `fail-fast` or any matrix strategy.

Important: Do NOT run release-it in the workflow. Release-it runs locally. The workflow only validates + uploads the tarball artifact to the Release that release-it already created.
